# EvoPi 实现阶段规约（Harness）

> 实现阶段的操作手册：**怎么防漂移、怎么防跨窗口失忆、怎么控 token/防大文件、怎么保证质量与可扩展**。
> 进度见 [进度.md](进度.md)（单一事实源）。设计契约在各模块方案文档 `../0X-*.md`。

---

## 0. 一句话

方案已全部定稿，实现阶段的唯一目标：**把方案变成能跑、可验收、可扩展的代码，且过程中不漂移、不失忆、不产生巨型文件。** 靠三样东西保证——**进度表（防漂移）+ hook（防失忆）+ 拆文件约定（控 token）**。

---

## 1. 防漂移：进度表是边界

- [进度.md](进度.md) 是实现的**单一事实源**。每模块拆 5 步（契约/接线/逻辑/命令/验收），每步一个勾。
- **任务边界 = 进度表里列的**。表里没有的、方案文档标 V2 的，一律不做（YAGNI）。想加范围先改进度表并记「变更日志」。
- 每个模块**先看方案文档的验收判据**，做到判据全绿才算 done——验收判据是防漂移的锚，不靠"感觉做完了"。

## 2. 防跨窗口失忆：hook 强制 + 记忆

跨窗口/压缩后不靠人记忆，靠机器：

- **`.claude/settings.json` 的 hook**：
  - `SessionStart` → 打印 impl/进度.md 的「当前模块 + 冻结约定」到上下文，新窗口一开就知道在哪、什么不能改。
  - `PreCompact` → 压缩前把「当前模块 + 未完成步骤」落一份快照，防压缩后忘进度。
- **记忆**：实现期的关键锁点（如「事件词表 v1 冻结」）写进 `~/.claude/.../memory/`，与现有记忆体系一致。
- **冻结约定**集中在进度表顶部「冻结的全局约定」一节——它是跨窗口最该被反复看见的东西。

## 3. 控 token / 防大文件：四条硬约定

1. **单文件软上限 ~500 行**。`evopi-trace/index.ts` 现 736 行已偏大 → 实现期**按模块拆子文件**（见 §5），`index.ts` 只做注册+路由。**边写边拆，不攒巨石**。
2. **大产出永不进上下文**。trace JSONL / 日志 / eval runs 只落 `.pi/evopi/`；读时用 Grep 或分段 Read，绝不整文件读进上下文。
3. **读文件优先分段**（已知看哪段只读哪段）——这是操作纪律。
4. **文档不与代码重复**。schema/挂点/逻辑在方案文档（设计契约），代码里放实现+注释，进度表只放「实现清单+验收」。**同一件事不写两遍**（正是这次把"每模块 impl 文档"砍掉的原因）。

## 4. 保证质量 & 可扩展：三个纪律

1. **契约先行、事件词表冻结**：动代码前，该模块的事件 schema/挂点已在方案文档定死并「冻结」。词表稳定是可扩展性地基（下游 SQLite、V2 自动化都依赖它不漂）。加字段 OK（向后兼容），改名/删字段要记变更日志。
2. **挂点写死、查证固化**：Pi 挂点以方案文档「已查证事实」为准（例：Top-K 不能挂 `context`，要挂 `resources_discover`——这类查证结论已固化在文档，实现时照做不臆测）。
3. **保守默认守底线**：总是拦高危、无 UI fail-safe、评分不猜、自动决策留 V2。安全 > 资源（tool_call 单 handler，模块 4 先于 5）。

## 5. 拆文件规范（控 token 的落地）

V1 所有模块**物理上仍在 `evopi-trace/` 扩展**（方案文档定的：先共存、后拆分），但**按模块拆成子文件**，避免 index.ts 继续膨胀：

```
.pi/extensions/evopi-trace/
  index.ts        # 只做：读配置 + 依次 registerXxx(pi) + 共享 tool_call 路由。目标 < 200 行
  trace.ts        # 底座：traceId / 事件写入 / JSONL / session anchor 工具函数
  cost.ts         # 模块2：registerCost(pi)
  memory.ts       # 模块3：记忆
  skill.ts        # 模块3：技能治理
  policy.ts       # 模块4/5 共享：读 policy.json + 风险判定
  job.ts          # 模块4：Job 信封 + checkpoint
  tools.ts        # 模块5：错误分类 + 预算 + 计时
  eval.ts         # 模块6：评分 + 子代理 + 棘轮
```

**约定**：
- 每个 `registerXxx(pi, shared)` 自注册钩子/命令，返回该模块要暴露给共享路由的 handler（如 policy/tools 的 tool_call 决策函数）。
- **共享 tool_call**：模块 4/5 不各注册 `tool_call`，而是把决策函数交给 index.ts 的单一 tool_call handler，内部按「安全→资源」顺序跑（兑现"安全>资源"、避免多 handler 顺序陷阱）。
- 事件写入统一走 `trace.ts` 的工具函数（一处控制 JSONL + anchor 判定），各模块不自己拼写 JSONL。
- `policy.json` 由 `policy.ts` 单点加载，模块 3/4/5 都问它——单一事实源。

**嵌套 CLAUDE.md（局部约定就近加载）——V1 暂不用，V2 拆目录时启用**：Claude Code 官方支持子目录放 CLAUDE.md，碰该子树文件时自动就近加载。**V1 不建**：所有模块挤在 `evopi-trace/` 一个目录，子目录 CLAUDE.md 覆盖的就是全部代码，与根 CLAUDE.md 无异且会与方案文档/进度表第三次重复同一批信息（违背"同一件事不写两遍"）。**V1 的等效替代**：把"本模块铁律"写在各 `.ts` 顶部注释（就近、跟代码走、改代码顺手改、零额外维护）。**启用触发条件**：当 V2 把扩展拆成 `evopi-cost/`/`evopi-policy/`/`evopi-eval/` 等**独立目录、各有不重叠的局部铁律**时，每目录放一份 CLAUDE.md 才名副其实（碰 policy 代码自动加载"改黑名单要同步 policy.json"、碰 cost 代码不被 policy 规则干扰）。与"V1 共存、V2 拆分"节奏对齐。

## 6. 每模块的实现流水线

```
① 冻结契约   —— 引方案文档的 schema/挂点/验收，不重写
② 拆子文件   —— 新建 <模块>.ts，写 registerXxx，index.ts 加一行注册
③ 核心逻辑   —— 只实现 V1 范围
④ 命令 / UX  —— /evopi-xxx
⑤ 自测验收   —— 跑命令 → grep JSONL 看预期事件 → 方案文档验收判据全绿
⑥ 提交       —— 中文 commit，小步（一个可验收单元一次）→ 勾进度表
```

## 7. 提交约定

- commit message 中文，小步（每个可验收单元一次，别攒大提交）。
- 在 `main` 上按现有工作方式提交（现状 branch=main）。
- 大产出物（JSONL/日志）确认已被 `.gitignore` 忽略，不进版本库。
