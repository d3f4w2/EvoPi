# 06 评测协作机制（Eval Collaboration Harness）最终方案

> 模块 6 / EvoPi 模块栈（最后一个核心模块）。状态：方案已定（2026-07-01），待实现。
> 遵循 [README.md](README.md) 文档规则：决策写理由 / 演化路线 / Pi 现状→EvoPi 增量→实现前后区别。
> 进度见 [进度.md](进度.md)。

## 一句话定位

加子代理协作 + 可测量的回归门禁——核心模型里「Eval = 对 Trace 的评分」的落地。它是**多处 V2 的共同依赖**（模块 3 自进化棘轮、模块 4 自动验收/自动 rewind 都要它提供评分+门禁）。做完 EvoPi 模块规划闭环。

**V1 = 评分引擎（确定性优先）+ 子代理（真做 spawn）+ 棘轮半自动**。全自动棘轮、LLM 评分、worktree 隔离推后。

### Pi 没有的（EvoPi 要补）

- **Turn/token 预算**：无 maxTurns → EvoPi 监听子进程 JSON 事件流的 message_end 计数、超阈值 kill。
- **JSON schema 约束**：无 responseFormat → EvoPi 用 prompt 指导 + 手工解析。
- **退出码可靠获取**：Pi 非零退出**抛错**、exitCode 藏在错误文本 → EvoPi 用 `命令; echo EXIT:$?` 解析。
- **worktree 隔离**：无内置 → 自己 `git worktree add` + cwd（V1 不做）。
- **评分引擎 / golden task / 棘轮**：Pi 完全无 → EvoPi 全建（现有 `/evopi-eval` 只 record score）。

### 实现前后区别

| 场景 | 实现前（Pi/evopi-trace） | 实现后（EvoPi Eval） |
| --- | --- | --- |
| 评估一次执行 | 无，靠人看 | golden task 确定性检查自动打分 |

## 二、决策（含理由）

### 决策 1：V1 = 评分引擎 + 真子代理 + 棘轮半自动

**选择**：确定性评分引擎 + golden task + **真做 spawn 评审子代理**（不退到单 agent 自评）+ 棘轮定框架但半自动（手动 replay + 人审）。

**为什么**：查证发现 Pi 子代理能力**现成且强**（完整 subagent 工具、进程隔离、工具/模型隔离、agent card 格式），所以「子代理」不必做轻——V1 就能真 spawn。评分引擎是地基（没有评分后面全免谈），确定性评分不难。全自动棘轮是「agent 改自己」最危险处，且要评分足够可信才敢无人裁决——V1 先半自动验证。Eval 是别的模块 V2 的依赖：V1 提供评分能力，让 V2 各模块自动化有裁决依据；Eval 自己的全自动棘轮也是 V2。

### 决策 2：确定性评分优先，LLM 评分只留接口

**选择**：主力用确定性检查（退出码/测试/文件变更/输出标记/禁区）；LLM 评分留接口、默认不开。

**为什么**：确定性检查客观、可重复、零成本、不幻觉——回归门禁的裁决必须可信，建在确定性上才敢用。LLM 评分补语义质量（可读性等），但每次多一次 LLM 调用（贵）、且可能不稳定，V1 确定性已够支撑回归门禁。默认不开，省钱不引不确定性（同 Cost）。

### 决策 3：参考 Pi subagent，spawn pi 子进程；多评估器架构支持默认单个

**选择**：参考 `examples/extensions/subagent` 实现，EvoPi 自己 spawn `pi --mode json --tools ... --model ...`；不 vendor 外部代码。多评估器（parallel spawn N 个投票）架构支持，V1 默认单评估器。

**为什么**：进程级隔离对「评估」最合适——评估子代理跑测试/改文件，独立进程 + 独立 cwd 天然隔离，不污染主 agent 的 session。参考而非 vendor，避免跟外部 example 的变更。parallel 模式让 darwin-skill 式「多评估器投票」能力架构上留好，但默认单个省 token，需高置信度（如棘轮门禁）才开多个。

### 决策 4：Agent card 复用 Pi frontmatter + 扩展字段

**选择**：name/description/tools/model/systemPrompt 用 Pi 原生，EvoPi 加 `maxTurns`（预算）+ `outputSchema`（期望输出）塞额外键，存 `.pi/evopi/agents/`。

**为什么**：同模块 3 决策 3 哲学——Pi frontmatter 允许额外字段，EvoPi 治理字段寄生其中，Pi 加载无视、EvoPi 自己读。一份文件复用 Pi 加载。turn 预算靠监听 message_end 计数执行；输出 schema 靠 prompt 指导 + 手工解析。**解析失败=记为无效评分，不猜**——确定性评分不能建在猜测上，宁可 invalid 不给假分，守 Eval 可信度底线。

### 决策 5：Golden task = YAML fixture，确定性 check，默认 all_pass，支持失败 trace 沉淀

**选择**：YAML 存 `.pi/evopi/evals/tasks/`，结构 `{id,description,prompt,checks[],scoring}`；check 都是确定性（command/file_changed/output_contains/no_forbidden）；评分粒度两者都支持、默认 all_checks_pass；**V1 支持从失败 trace 生成 task 候选**。

**为什么**：YAML 可读可 diff，适合人写 golden task。all_pass 二值对回归门禁够用，weighted 留细粒度。「失败 trace 变 task」兑现「从 Trace 沉淀资产」哲学、与 stub「失败 trace 可变 test」一致——模块 4 的失败 job/被拦高危一键生成负例（人审后入库），形成闭环。

### 决策 6：退出码用 echo $? 可靠拿；评分绑 trace/task/dataset id

**选择**：command check 跑 `命令; echo EXIT:$?` 从输出解析；评分记录绑 `traceId + taskId + datasetId`。

**为什么**：Pi 把非零退出转成异常、exitCode 藏错误文本、格式可能变——用 echo $? 最朴素可靠、跨平台一致。绑三个 id 让评分可追溯（回到哪次执行、哪个 task、每个 check 结果），是回归门禁的证据基础（stub 要求）。

### 决策 7：棘轮半自动，不降即可但需人审，全量跑 golden task

**选择**：候选→安全扫描（复用 policy）→`/evopi-eval replay` 手动触发→展示 baseline vs candidate→人拍板 enable→记回滚元数据。门禁=**不降即可（持平也过）但需人审**；replay **全量跑 golden task**。

**为什么**：棘轮=只进不退，防模型自作主张的改动悄悄变差。V1 半自动（人在环）——全自动 enable 是「agent 改自己」最危险处，要评分足够可信才敢放开（同模块 3/darwin-skill 人在环）。「不降即可」对齐 stub「改善或保持」——新能力可能不提分但有价值（如多覆盖一个场景）；人审兜底。全量跑防「改 A 弄坏 B」（回归本意）。基线=候选要替换的现有资产分/历史最佳。

### 决策 8：独立 `eval.*` 事件，gate/candidate 写 session，闭环 skill.accepted

**选择**：`eval.score`（已有）/`eval.run`/`eval.replay`/`eval.gate`/`eval.candidate`。**写 session anchor**：`eval.gate`（门禁裁决）+ `eval.candidate`（产生 task）；**只进 JSONL**：eval.score/run/replay。

**为什么**：套用 3.6 判据——关键决策/资产变更写 session，逐次观测只 JSONL。门禁裁决（enable/reject 候选）和 `skill.approved` 同级、必须可追溯；评测过程量大只进 JSONL。**闭环点**：`eval.gate` enable 就是资产被接受的时刻，可触发 `skill.accepted`/记忆生效——模块 3（技能候选）与模块 6（评分裁决）在此接上，兑现接力上下文最早提的 `evopi.skill.accepted`。

## 三、实现

### 钩子与挂点

| 用途 | 挂点 | 备注 |
| --- | --- | --- |
| spawn 评估子代理 | spawn `pi --mode json` | 参考 Pi subagent |
| turn 预算 | 监听子进程 JSON 事件流 message_end | 超 maxTurns kill |
| 确定性评分 | 解析子代理输出 + `echo $?` | command/file/marker/禁区 |
| 命令/展示 | `registerCommand` | `/evopi-eval` 系列 |
| 事件 | `pi.appendEntry` + JSONL | eval.* |

### Golden task 结构（YAML）

```yaml
# .pi/evopi/evals/tasks/fix-login-bug.yaml
id: fix-login-bug
description: 修复登录超时 bug
prompt: 修复 src/auth.ts 里登录超时的问题
checks:
  - { type: command, run: "npm test -- auth", expect: exit_zero }
  - { type: file_changed, path: src/auth.ts }
  - { type: output_contains, marker: "登录成功" }
  - { type: no_forbidden, paths: [".env", "package.json"] }
scoring: all_checks_pass    # 或 weighted
```

### Agent card（复用 Pi frontmatter + 扩展）

```yaml
---
name: eval-scorer
description: 评估执行结果并打分
tools: bash, read, grep          # Pi 原生
model: claude-haiku-4-5          # Pi 原生
maxTurns: 5                      # EvoPi 扩展：turn 预算
outputSchema: { score: number, reason: string, passed: boolean }  # EvoPi 扩展
---
你是评估子代理。跑测试、检查文件变更，用 `命令; echo EXIT:$?` 取退出码，按 outputSchema 输出 JSON。
```

### 评分记录

```jsonc
{ evalId, taskId, traceId, datasetId, score, passed,
  checkResults: [{ type, passed, detail }], timestamp }
```

### 命令集

- 已有：`/evopi-eval record <name> <score> [notes]`、展示
- 新增：`/evopi-eval run <task>`（跑 golden task）、`/evopi-eval replay <candidate>`（棘轮 replay）、`/evopi-eval gate <candidate>`（门禁裁决）

### 实现落点

V1 复用 `.pi/extensions/evopi-trace/index.ts`（`/evopi-eval` MVP 已在其中）。模块拆分后迁到 `.pi/extensions/evopi-eval/`。

## 五、架构优点

1. **闭环整个 EvoPi**：Eval 是模块 3/4 自动化的裁决引擎，`eval.gate` enable 接上 `skill.accepted`——「从 Trace 沉淀资产→评分门禁→受控启用」全链路打通。
2. **复用 Pi 强子代理**：子代理/工具模型隔离/agent card Pi 全现成，V1 就能真 spawn 评审子代理，增量集中在评分/棘轮。
3. **评分可信**：确定性优先 + echo $? 可靠退出码 + 解析失败不猜——回归门禁建在可信评分上。
4. **受控自进化的安全闸**：棘轮半自动（不降即可 + 人审 + 全量 replay），既防悄悄退步，又留人在环，避免「agent 改自己」失控。
5. **失败变资产**：失败 trace 沉淀成 golden task 负例，防同类回归——EvoPi「证据链」哲学的收尾。

---

## 六、少量缺点

1. **子进程开销**：每次评估 spawn 独立 pi 进程，比同进程慢、耗资源——换来的是彻底隔离，评估场景可接受。
2. **输出 schema 靠 prompt**：Pi 无 responseFormat，子代理可能不严格按格式输出→记无效评分（不猜），偶尔浪费一次评估。
3. **golden task 要人建**：V1 主要靠人写 golden task（失败 trace 生成是候选、仍要人审），初期覆盖度有限。
4. **棘轮半自动**：V1 每次候选启用要人拍板，模块 3/4 的全自动自进化要等 V2——V1 阶段自进化是「半自动化」。
5. **确定性覆盖不了语义**：V1 不开 LLM 评分，「代码好不好」这类软质量评不了，要等 V2。
6. **无 worktree 隔离**：V1 评估子代理直接在工作目录跑（靠受保护路径软防护），并发/破坏性评估要等 V2 的 worktree。
