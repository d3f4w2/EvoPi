# 04 执行治理机制（Execution Harness）最终方案

> 模块 4 / EvoPi 模块栈。状态：方案已定（2026-07-01），待实现。
> 遵循 [README.md](README.md) 文档规则：决策写理由 / 演化路线 / Pi 现状→EvoPi 增量→实现前后区别。
> 进度见 [进度.md](进度.md)。

## 一句话定位

把用户请求变成「受治理的工程 job」——协调 plan / act / review / 审批 / checkpoint / 验收，而不是让工具调用作为互不相关的事件散落执行。这是 EvoPi 第一个**主动介入执行**（拦截高危、要审批）的模块，前三个模块（Trace 看 / Cost 省 / Memory 记）都是旁路的。

**V1 灵魂 = Policy Gate**：拦危险操作 + 高危要审批。这是面试一眼懂价值、调研列 P0 的能力。

## 一、Pi 现状 → EvoPi 增量 → 实现前后区别

> 本节回答「Pi 已实现什么、我们还要实现什么、实现前后有何区别」。所有 Pi 能力均已查源码确认（见记忆 evopi-execution-pi-capabilities）。

### Pi 已经实现的（白送的底座）

| 能力 | Pi 怎么实现的 | 源码位置 |
| --- | --- | --- |
| **工具调用可拦截** | `tool_call` handler 返回 `{block:true, reason}` → agent-loop 把它转成**错误工具结果**返回给模型（`createErrorToolResult(reason)`, isError:true）。模型看到「工具失败+原因」，不是静默跳过。 | `agent-loop.ts:581-607`、`runner.ts:862-883` |

| **拦截早退** | 多扩展按加载顺序跑，第一个返回 block 立即 return，后续不跑。 | `runner.ts:866-878` |
| **项目级信任** | `project_trust` 钩子（yes/no/undecided）+ `ctx.isProjectTrusted()`——全局「本项目可否执行危险操作」开关。 | `types.ts:503-525` |
| **交互式 UI** | `ctx.ui` 提供 `confirm(title,msg)→bool`、`select`、`input`、`custom` 等，不止 notify。 | `types.ts:124-275` |

| **输入拦截** | `input` 钩子在用户输入时触发，可 `{action:transform/handled/continue}` 改写或拦截。 | `types.ts:792-808` |
| **Job 命令 MVP** | `/evopi-job` 已实现 start/plan/acceptance/状态切换 + `evopi.job` 条目 + `job.*` 事件。 | `.pi/extensions/evopi-trace/index.ts:469-548` |

### Pi 没有的（EvoPi 要补的缺口）

- **工具级审批**：Pi 只有项目级 `project_trust`（全局开关），**没有**「单次高危操作弹确认」机制 → EvoPi 用 `ctx.ui.confirm()` 在 `tool_call` handler 里自己实现。
- **风险分级 / Policy Gate**：Pi 的 `tool_call` 是裸钩子，不判断「这次调用危不危险」→ EvoPi 加三级风险判定 + 黑名单/受保护路径扫描。
- **自动 rewind**：Pi 有 `navigateTree`（能回退），但**不会自动**检测失败并回退 → EvoPi V1 做手动 rewind，自动留 V2。
- **基于证据的验收**：Pi 无验收概念 → EvoPi 汇总 trace 证据做辅助判定。
- **job 与执行的联动**：现有 job 是被动状态记录器，拦截/审批/checkpoint 还没喂进它 → EvoPi 补联动。

### 实现前后区别（做了这个模块，行为变了什么）

| 场景 | 实现前（现在的 Pi/evopi-trace） | 实现后（EvoPi Execution） |
| --- | --- | --- |
| agent 要跑 `rm -rf /data` | 直接执行 | 🛑 被 Policy Gate 拦截，弹审批；无 UI 时直接 block |
| agent 改文件出错后 | 继续，用户手动发现 | 危险写操作前已打 checkpoint，可 `/evopi-job rewind` 回到改动前 |
| 一次任务的执行 | 一堆散落的 tool_call | 被包成一个 job，plan/证据/审批/checkpoint 都挂在 job 上 |
| 判断任务完成 | 看 agent 说「我做完了」 | 看证据面板：tool_error 数、测试过没过、高危有无未解决 |
| 审批/拒绝记录 | 无 | `policy.approved`/`denied` 写 session，可追溯 |

## 二、决策（含理由）

### 决策 1：介入强度 = 主动拦截（高危真 block + 审批）

**选择**：Policy Gate 在 `tool_call` 里对高危操作真 block，要用户审批才放行。

**为什么**：这是本模块与前三个（旁路观测）的本质区别，也是其核心价值。查证确认 Pi 的 `tool_call` 能真 block（转错误结果给模型），技术上无障碍。若只观测告警不拦截，「治理」就退化成「旁观」，和 Trace/Cost 没本质区别。

### 决策 2：Job 信封不大改，Job 与 Trace 一对多

**选择**：现有 `/evopi-job` 信封只补 3 点（与 Policy Gate 联动 / `turn_end` 记 entryId 供 rewind / plan-act-review 用现有字段表达，不加 phase 状态机）；Job 手动 `/evopi-job start` 创建，一个 job 跨多轮 trace。

**为什么**：现有信封已能 resume、绑 traceId、记成败，满足 stub「job 可 resume」——重新设计是浪费。Execution 的价值在 Policy Gate 不在信封。一对多是因为「一个完整任务」通常跨多轮 provider 请求，手动 start 避免把闲聊/查询也包成碎屑 job。

### 决策 3：风险三级 + 总是拦高危（与项目信任无关）

**选择**：low（只读）放行 / medium（写文件）放行+记 trace / high（黑名单+受保护路径）block+审批。**高危无论 `isProjectTrusted()` 是否为真都拦。**

**为什么**：「项目受信任」只决定「能不能启动这个项目」，**不等于**「可以 rm -rf」——这是两个不同语义，混用会让最危险的操作在受信任项目里裸奔。治理模块的保守默认必须守住高危这条线。

### 决策 4：危险动作定义抽成共享 policy，模块 3/4 都读

**选择**：黑名单（`rm -rf`/force-push/publish/`curl|sh`/secrets）+ 受保护路径清单，抽成一份共享定义（如 `.pi/evopi/policy.json` 或代码常量）。

**为什么**：兑现模块 3 决策 9 的协调点。模块 3 静态扫 skill 文本、模块 4 运行时扫实际 tool_call，是两道防线；但若各自维护黑名单，两套定义必然漂移不一致。单一事实源避免这个坑。

### 决策 5：审批内联门禁，无 UI = fail-safe，记住选择要收紧

**选择**：high 判定后先 `ctx.ui.confirm()` → 批准放行 / 拒绝 block。**无 UI 模式（hasUI=false）一律 block（fail-safe）**。审批**支持「本会话都批准这个命令」**，但收紧：①「记住」只对**具体命令指纹**生效（`rm -rf /tmp/x` 记住 ≠ 所有 rm 放行）；②最危险的 blocked 级不允许记住，每次都问。

**为什么**：无 UI 时弹不出确认，保守起见按「总是拦高危」的逻辑一律 block。「记住选择」是便利性妥协，但若作用域太宽，一次误批会放行整个会话的危险操作——所以按命令指纹收紧、且最危险级禁止记住，在便利和安全间取平衡。

### 决策 6：checkpoint 仅危险变更前打，rewind V1 只做手动

**选择**：checkpoint = `ctx.setLabel` 打标签（零成本，Pi 已有 entry）+ 存进 job checkpoint 列表，**仅 medium/high 写操作前自动打** + 手动 `/evopi-job checkpoint`。rewind = **V1 只做手动** `/evopi-job rewind <checkpoint>`；自动 rewind 推 V2。

**为什么**：checkpoint 打在「危险写操作前」，回退点都是干净点，语义闭环且少噪声（每 turn 都打是重复，Pi 本就有 entry）。自动 rewind 的判定难题——不是所有 tool_result 失败都该回退（有些失败模型会自己重试），误触发比不回退更糟；且「何为该回退的失败」本质要 Eval 判定，故推 V2。

### 决策 7：验收 = 证据面板 + 辅助判定，人拍板

**选择**：`/evopi-job` 汇总 trace 证据（tool_call/error 数、高危拦截、测试是否通过）；用户切 passed 时若有 error/高危未解决则警告。**测试证据靠启发式识别测试命令（npm test/pytest/go test）+ tool_result 的 isError**。自动裁决留 V2。

**为什么**：兑现 stub「acceptance 基于证据而非最终文本」。但「满足验收标准」是语义判断，无 Eval 模块硬做自动裁决易误判（把有 error 的判 passed）——不如把证据摆清楚让人拍板可靠。与 EvoPi「V1 观测、V2 自动化」节奏一致。

### 决策 8：独立 `job.*`/`policy.*` 事件，套用 3.6 session anchor 判据

**选择**：新增独立事件不污染 Trace core。**写 session anchor**：审批决策（`policy.blocked`/`approved`/`denied`）+ checkpoint/rewind + job.start/终态。**只进 JSONL**：`policy.check`（逐次风险判定），且 `policy.check` 只记 medium/high、low 不记。

**为什么**：套用模块 3.6 确立的判据——关键治理决策/资产变更写 session，逐次观测只进 JSONL。审批和 checkpoint 是 session 级治理证据链（stub 要求审批可追溯），逐次风险判定是执行细节。low（只读）量最大且无治理价值，不刷 trace。

## 三、实现

### 钩子与挂点（均已查证）

| 用途 | 挂点 | 备注 |
| --- | --- | --- |
| Policy Gate 拦截 | `tool_call`（返回 `{block,reason}`） | 判定 + 审批内联于此 |
| 审批交互 | `ctx.ui.confirm()` | 无 UI 时 fail-safe |
| checkpoint 打标 | `tool_call`（危险写前）+ `ctx.setLabel` | 存进 job |
| rewind | `/evopi-job rewind` + `ctx.navigateTree` | 手动 |
| 验收证据 | `tool_result`（isError + 测试命令识别） | 汇总进 job |
| Job 包装 | `input` 钩子（可选） / `/evopi-job start` | V1 用命令即可 |
| 事件 | `pi.appendEntry` + JSONL | job.*/policy.* |

### Policy Gate 判定流程

```text
tool_call 触发
  → 判定 riskLevel（按工具类型 + 内容扫描共享 policy 定义）
  → low  : 放行（不记 policy.check）
  → medium: 放行 + 记 policy.check(JSONL)
  → high : 打 checkpoint → ctx.ui.confirm()
             ├ 有 UI + 批准 → 放行 + policy.approved(session)
             ├ 有 UI + 拒绝 → {block,reason} + policy.denied(session)
             └ 无 UI       → {block} + policy.blocked(session, "无法审批")
```

### 共享 policy 定义（模块 3/4 共用）

```jsonc
// .pi/evopi/policy.json（或代码常量）
{
  "dangerousCommands": ["rm -rf", "git reset --hard", "git push --force", "curl|sh", "npm publish"],
  "protectedPaths": [".git/", ".env", "secrets/", "~/.ssh/"],
  "testCommandPatterns": ["npm test", "pytest", "go test", "cargo test", "vitest"]
}
```

### 命令集

- 已有：`/evopi-job`、`start`、`plan`、`acceptance`、状态切换
- 新增：`/evopi-job checkpoint`、`/evopi-job rewind <checkpoint>`

### 实现落点

V1 复用 `.pi/extensions/evopi-trace/index.ts`（`/evopi-job` MVP 已在其中）。模块拆分后迁到 `.pi/extensions/evopi-execution/`。

| 版本 | 范围 | 说明 |

## 五、架构优点

1. **真治理，非旁观**：第一个能真正拦住危险操作的模块，`tool_call` block 已验证可行，面试价值直白（安全焦虑刚需）。
2. **复用 Pi、缺口精准**：拦截/UI/checkpoint 全用 Pi 原生能力，只补 Pi 缺的「风险判定 + 工具级审批 + 自动化」，风险面小。
3. **保守默认守底线**：总是拦高危、无 UI fail-safe、记住选择收紧——安全的默认姿态一致贯穿。
4. **证据链完整**：审批/拒绝/checkpoint 写 session anchor，可追溯、可重建，延续 EvoPi「证据链」哲学。
5. **协调不冲突**：与模块 3 共享 policy 定义（静态+运行时两道防线），与后续 Eval/Tool Runtime 留好联动位。

## 六、少量缺点

1. **风险判定靠启发式**：黑名单+路径扫描可能漏判（新型危险命令）或误判（含 rm 但安全的场景），需持续维护 policy 定义。
2. **审批打断心流**：高危每次弹确认（除非记住），交互式使用时可能烦——换来的是不误执行危险操作。
3. **无 UI 场景受限**：fail-safe 会挡住自动化/CI 流程的高危操作，需白名单或显式配置豁免（V1 的白名单尚未展开）。
4. **验收不自动**：V1 只摆证据不裁决，「测试过没过」靠启发式识别可能不全（非标准测试命令识别不到）；真正的自动验收要等 V2 + Eval。
5. **rewind 是手动的**：出错后要用户主动发现并回退，自动自愈留 V2。
6. **checkpoint 非真快照**：基于 session tree entry 的标签，回退的是「对话/上下文」，不回退**文件系统**改动——文件层面的回滚需配合 git 或 V3 沙箱能力。
