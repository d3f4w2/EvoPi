# 05 工具运行时机制（Tool Runtime Harness）最终方案

> 模块 5 / EvoPi 模块栈。状态：方案已定（2026-07-01），待实现。
> 遵循 [README.md](README.md) 文档规则：决策写理由 / 演化路线 / Pi 现状→EvoPi 增量→实现前后区别。
> 进度见 [进度.md](进度.md)。

## 一句话定位

治理外部工具与执行后端——工具的**可靠性与资源**。与模块 4 分工：**模块 4 是「保安」（该不该放行），模块 5 是「运维」（跑得好不好、花多少、多快）**。

**V1 = 工具可观测性 + 预算**：失败归一化分类、工具预算、延迟/可靠性统计。MCP broker、沙箱后端、浏览器验收推后。

## 一、Pi 现状 → EvoPi 增量 → 实现前后区别

> Pi 能力均已查源码确认（见记忆 evopi-tool-runtime-pi-capabilities）。**本模块 Pi 底座几乎全空白**——空间大，但要全自建。

### Pi 已经实现的

| 能力 | Pi 怎么实现的 | 源码位置 |
| --- | --- | --- |
| **工具错误（二值）** | 失败都打包成 `createErrorToolResult(message)`，`tool_result.isError=true`。**只有布尔，无分类**。 | `agent-loop.ts:659-665,716-721` |
| **bash 超时** | Shell.exec 支持秒级 timeout，超时 `killProcessTree` 并返回 `ExecutionError("timeout")`。**仅 bash，工具本身无超时**。 | `nodejs.ts:312-363` |
| **多 handler 全执行** | 多个 `tool_call` handler **全部执行**（for 循环非早退），最后一个非 undefined 结果生效。`ToolCallResult={block?,reason?}`。 | `agent-harness.ts:249-266` |
| **动态工具启停** | `setTools`/`setActiveTools(names)` 可动态替换工具集或改 active 列表（但只能整体启停，无能力粒度）。 | `agent-harness.ts:906-963` |
| **本机执行** | 工具经 `ctx.exec` → Node `spawn()` 在**本机直接执行**，有 AbortSignal + timeout，**无容器/隔离**。 | `nodejs.ts:250-373` |

### Pi 没有的（EvoPi 要全自建）

- **错误分类**：Pi 只给 isError 布尔 → EvoPi 从 `tool_result.content[0].text` 启发式推断 7 类。
- **延迟**：`tool_result` **无时间戳** → EvoPi 自己在 tool_call 记 start、tool_result 算差。
- **工具级超时**：只 bash 有 → EvoPi 给每个工具建带 timeout 的 AbortController。
- **预算/限额**：Pi 完全无工具次数/花费限制 → EvoPi 在 tool_call 自建计数器 + block。
- **MCP**：Pi **零 MCP 基础** → 完全新增模块（V1 不做）。
- **权限画像 / 沙箱**：无 → V1 用共享 policy 的受保护路径做轻量防护，真沙箱推后。

### 实现前后区别

| 场景 | 实现前（Pi/evopi-trace） | 实现后（EvoPi Tool Runtime） |
| --- | --- | --- |
| 工具失败 | 一个 isError 布尔 | 归一化为 timeout/auth/permission/… 7 类 |
| 工具卡死 | 非 bash 工具无限等待 | 工具级超时 abort + 归 timeout 类 |
| agent 死循环调工具 | 一直跑 | 超预算次数 → block「预算耗尽」 |
| `/evopi-tools` | 裸 call/error 计数 | 「bash: 45 次/3 错(2 timeout)/均 1.2s/最慢 8s」 |
| 工具多慢 | 无从得知 | tool.result 带 latencyMs |

## 二、决策（含理由）

### 决策 1：模块 5 = 可靠性/资源，与模块 4（安全准入）分工但共享 tool_call

**选择**：职责分开——模块 4 管「该不该放行」，模块 5 管「跑得怎么样/花多少」；但都挂 `tool_call`。

**为什么**：两者关注点不同（安全 vs 可靠性），概念上该分开，避免混为一谈。查证确认多个 tool_call handler 全部执行，所以能共享挂点。V1 范围：错误分类 + 预算 + 延迟统计 + 受保护路径（复用模块 4 共享 policy）。

### 决策 2：失败归一化 7 类，启发式推断，内置规则表

**选择**：从 `tool_result.content[0].text` + isError 启发式推断 `timeout/auth/schema/permission/rate_limit/tool_bug/unknown`；关键词规则内置代码、V1 不可配。

**为什么**：Pi 无结构化错误码，只能从错误文本推断。**诚实局限**：关键词匹配不可能 100% 准，会漏判/误判——但即使 70% 准，也比一个 isError 布尔强得多，能支撑「这工具老 timeout」这类可靠性洞察。可配置规则（用户自定义关键词）留 V2，V1 内置够用。

### 决策 3：V1 自建工具级超时

**选择**：tool_call 里给每个工具建带 timeout 的 AbortController，超时 abort + 归 timeout 类。

**为什么**：Pi 只有 bash 超时，非 bash 工具（未来 MCP/自定义）卡死无法中断。自建工具级超时补上这个缺口，是可靠性基础；且超时本身是一种「资源耗尽」，和预算体系一致。这个「工具执行包装层」也顺便探出了 V2 沙箱的扩展点。

### 决策 4：预算三维计量，分工具 + 全局，硬限只卡次数

**选择**：计量调用次数 / 累计延迟 / 错误次数；层级 per-turn（防单轮死循环）+ per-job（存进 job，与模块 4 联动）；粒度**分工具计数 + 全局总预算**；**硬限（block）只卡调用次数**，延迟/错误超阈值只软限告警。

**为什么**：分工具能定位「哪个工具失控」并对不同工具设不同上限（read 宽、bash 严），全局总预算是总闸。硬拦只用在「调用次数」这个最明确的失控信号上——延迟/错误的阈值判定模糊，硬拦易误伤正常的慢工具或会自愈的重试。软限告警覆盖面广、不误伤。默认示例：per-turn 单工具 50、per-job 单工具 200、软限 80%。

### 决策 5：延迟统计只算 avg+max+count

**选择**：自建计时（tool_call 记 start、tool_result 算差、按 toolCallId 配对）；扩展 toolStats 加 `errorsByClass` + `latency{count,totalMs,maxMs}`；V1 只算简单统计，p50/p95 留 V2。

**为什么**：avg+max 内存 O(1)，已能看出「工具慢不慢、有无极端慢」，够 V1 可靠性洞察。精确分位数要样本窗口或 t-digest，更重，价值不足以进 V1。

### 决策 6：单 handler 统一决策，安全 > 资源

**选择**：V1 模块 4/5 同在 evopi-trace 扩展，用**一个** tool_call handler，内部按优先级跑：先 policy 安全准入 → 再预算资源，谁先命中 block 用谁 reason。

**为什么**：查证发现多 handler「最后结果生效」会带来顺序陷阱——某工具既高危又超预算时，block 原因取决于注册顺序（偶然）。单 handler 统一决策让原因**确定**：安全 > 资源，因为 `rm -rf` 比「次数超限」严重，用户应先知道是危险被拦。V2 拆成独立扩展后靠注册顺序约定（policy 后注册覆盖）。

### 决策 7：V1 不做 MCP/沙箱，留通用接口位

**选择**：不做 MCP broker / 沙箱后端 / 浏览器验收。事件**预留通用字段（toolName/errorClass/latency/source），不预先建 mcp.*/sandbox.* 专用事件**。

**为什么**：MCP 是 Pi 零基础的独立大模块，沙箱是重型工程（OpenSandbox 仅设计参考）——都不该塞进 V1。且 V1 的错误分类/预算/延迟**对所有工具统一生效**（都走 tool_call），V2 加 MCP 时 MCP 工具自动享受这套治理、无需改 V1。通用字段 + source 区分来源，满足 stub「将来不改词表」验收；不为没实现的东西提前建空事件。

### 决策 8：扩展 tool.result 加客观属性，预算独立成 tool.budget

**选择**：`errorClass`/`latencyMs` 加进现有 `tool.result` 事件；预算告警用独立 `tool.budget`。

**为什么**：确立**新判据补充「独立事件」原则**——errorClass/latency 是工具执行的**客观属性**（任何观测者都想知道多久/错哪类），加进 tool.result 是 Trace 增强不是污染；而预算是模块 5 的**治理语义视角**（和 policy 一样），该独立。**session anchor**：tool.result 只进 JSONL（逐次观测量大），tool.budget 硬限 block 写 session（因预算拦截执行，同 policy.blocked 级）。

## 三、实现

### 钩子与挂点

| 用途 | 挂点 | 备注 |
| --- | --- | --- |
| 计时开始 + 预算/policy 判定 | `tool_call`（单 handler 统一决策） | 安全>资源；超预算 block |
| 工具级超时 | `tool_call` 建 AbortController | 超时 abort |
| 计时结束 + 错误分类 + 统计 | `tool_result` | 算 latency、推断 errorClass、更新 toolStats |
| 预算/统计展示 | `/evopi-tools` | 已有，扩展 |
| 事件 | `pi.appendEntry` + JSONL | tool.result 扩展 + tool.budget |

### 错误分类规则表（内置示例）

```text
timeout    ← "timeout" / "timed out" / 自建超时 abort
auth       ← "401" / "unauthorized" / "authentication" / "api key"
permission ← "403" / "permission denied" / "EACCES" / "forbidden"
schema     ← "invalid argument" / "validation" / 参数校验失败
rate_limit ← "429" / "rate limit" / "too many requests"
tool_bug   ← 异常栈特征，不属以上
unknown    ← 兜底
```

### 工具统计结构

```jsonc
toolStats[toolName] = {
  calls, errors,                              // 已有
  errorsByClass: { timeout: 3, permission: 1 },   // 新
  latency: { count, totalMs, maxMs },             // 新（avg=totalMs/count）
  lastUsedAt
}
```

### 预算结构（per-turn / per-job）

```jsonc
budget = {
  perTool: { bash: {limit:200, used:...}, read: {limit:1000, used:...} },
  global:  { limit: 500, used: ... },
  softLimitPercent: 80
}
```

### 命令集

- 已有：`/evopi-tools`（call/error 计数）
- 扩展展示：错误分类分布、延迟 avg/max、预算用量

### 实现落点

V1 复用 `.pi/extensions/evopi-trace/index.ts`（`/evopi-tools` + tool_call/tool_result 钩子已在其中；与模块 4 共用同一 tool_call handler）。模块拆分后迁到 `.pi/extensions/evopi-tools/`。

## 五、架构优点

1. **补底座大片空白**：Pi 工具运行时几乎全空白（错误/延迟/预算/MCP/沙箱都无），本模块补上，增量价值明显。
2. **可靠性可量化**：错误分类 + 延迟统计把「工具跑得怎么样」变成数据，为 V2 的智能路由/降级铺路。
3. **防失控**：预算硬限（次数）+ 工具级超时，挡住 agent 死循环/卡死，安全网直接。
4. **与模块 4 干净协作**：单 handler 统一决策（安全>资源），共享 policy 定义，不重复挂钩、不顺序碰运气。
5. **演进不返工**：治理对所有工具统一生效 + 通用事件字段，V2 加 MCP/沙箱平滑接入、不改词表。

---

## 六、少量缺点

1. **错误分类靠启发式**：从文本推断不可能全准，非标准错误信息会漏判/误判——V1 接受，V2 可配置改善。
2. **延迟只有 avg/max**：会被极端值影响，看不出分布，p50/p95 要等 V2。
3. **预算默认值需调**：per-turn 50/per-job 200 是拍的初值，不同项目/工具合理值不同，要实践中校准。
4. **无真隔离**：V1 只有受保护路径（路径级软防护），工具仍在本机跑，真沙箱要 V2/V3——高危隔离场景 V1 覆盖不了。
5. **超时可能误杀**：工具级超时若设太短，会中断正常的长任务（如大型构建）——超时阈值要合理，或按工具区分。
6. **统计不跨会话**：与 Cost 一样，工具统计 in-memory + JSONL，跨会话趋势要等 SQLite（跟 Trace 走）。
