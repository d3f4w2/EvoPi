## 当前模块详情：Context Cost Harness（模块 2）

**定位**：主动优化模块，但 V1 只做「缓存观测 + 预算观测」，不裁剪历史、不自动 compact、不做 RAG。

### 子议题清单

| # | 子议题 | 状态 | 结论摘要 |
| --- | --- | --- | --- |
| 2.0 | 缓存这半的整体定位 | 🟡 方案已定 | 选 **A+B**：A=读真实 usage 算缓存命中率写 trace + `/evopi-cost` 展示；B=显示 retention 档位并在合适时**建议**切 long。**C（主动注入 cache_control）砍到 V2**——因为 Pi 已自动用掉 3/4 个 Anthropic 断点，手动注入重复且会挤掉 Pi 的。 |
| 2.1 | `/evopi-cost` 命令形态 | 🟡 方案已定 | **两个命令**：`/evopi-cost`（摘要快照：上下文压力+档位、缓存命中率、本会话累计 token/成本、请求数）；`/evopi-cost detail`（原始 usage / payload / trace path）。**告警**：上下文压力 ≥80/90/95% 时 `ctx.ui.notify` 主动提醒 + 写 `cost.pressure` 到 trace；**不自动 compact、不自动切 retention**。 |
| 2.2 | Cost 写 Trace 的事件结构 | 🟡 方案已定 | **新增 3 个独立事件 `cost.request` / `cost.cache` / `cost.pressure`，不动 `provider.*`**（Trace core 不被 Cost 语义污染，符合「Trace 底座 / Cost 消费者」）。`cost.request`=请求侧（contextEstimate/retention/msg+tool count）；`cost.cache`=响应后读 leafEntry.usage（input/output/cacheRead/cacheWrite/cacheWrite1h + cacheHitRate + cost）；`cost.pressure`=跨阈值时一次（level/percent/tokens）。`cost.*` 只写 JSONL，不写 session anchor（符合 Anchor-only）。 |

| 2.4 | 是否需要 Cost 独立 SQLite | 🟡 方案已定 | **V1 只 JSONL + in-memory，不建独立 SQLite**。in-memory 累加供 `/evopi-cost` 即时读；`cost.*` 写 JSONL。不自立门户，将来 Trace 上 SQLite 时 `cost.*` 复用同一 store。理由：与 Trace「JSONL only/SQLite 后置」一致；V1 只需本会话快照+告警，无需跨会话查询（YAGNI）。 |

### 已查清的 Pi 缓存事实（决策依据，源码已确认）

- Pi 在 Anthropic 上**自动**打 `cache_control`，覆盖 ①每个 system text 块 ②最后一个 tool 定义 ③最后一条 user 消息的最后一个块；Anthropic 总共只允许 4 个断点，Pi 已占 3 个。
- 由 `cacheRetention` 控制：`none` / `short`(5m 默认) / `long`(1h，模型支持时)，env 变量 `PI_CACHE_RETENTION=long`。
- `before_provider_request` 的 `event.payload` 是裸 Anthropic 请求体，handler 返回值能真正改写并发给 provider。
- `after_provider_response` 事件**不含 usage**；真实 token（`cacheRead`/`cacheWrite`/`cacheWrite1h`）在 `ctx.sessionManager.getLeafEntry()` 的 assistant message `.usage` 里。
- `getContextUsage().tokens` 是**估算**，compaction 后到下次响应前为 null。

## 当前模块详情：技能记忆机制（模块 3）

## 当前模块详情：执行治理机制（模块 4）

**定位**：把用户请求变成「受治理的工程 job」——协调 plan / act / review / 审批 / checkpoint / 验收，而不是让工具调用作为互不相关的事件散落执行。这是 EvoPi 第一个会**主动介入执行**（拦截/审批）的模块。

### 子议题清单

| # | 子议题 | 状态 | 结论摘要 |
| --- | --- | --- | --- |
| 4.0 | 模块定位与 V1 范围 | 🟡 方案已定 | **介入强度=主动拦截**：Policy Gate 在 `tool_call` 里对高危操作真 block + 要审批（这是模块灵魂，已查证 Pi 原生支持 block）。**V1 范围**：①Job 信封（已实现，完善）②Policy Gate（核心，新建）③审批（ctx.ui.confirm）④checkpoint（Pi 原生 fork/navigate，纳入 V1）⑤验收轻量版。**自动 rewind**（检测失败自动回退）要 EvoPi 自己做，列 V1 进阶/V2。 |
| 4.1 | Job 信封模型（状态机/与 trace 绑定） | 🟡 方案已定 | 现有 `/evopi-job` 信封**不大改**，只补 3 点：①与 Policy Gate 联动（拦截/审批时更新 job 状态）②`turn_end` 把 session entryId 记进 job（供 rewind）③plan/act/review 用现有字段表达（plan=plan 字段、act=toolCalls/toolErrors、review=acceptance），**不加 phase 状态机**。**Job 与 Trace 一对多**：手动 `/evopi-job start`，一个 job 跨多轮 trace（把完整任务包成 job，不产生碎屑 job）。理由：现有信封已能 resume/绑 traceId/记成败，重设计是浪费；Execution 价值在 Policy Gate 不在信封。 |
| 4.2 | Policy Gate：风险分级与拦截 | 🟡 方案已定 | **三级**：low（只读 read/grep/find/ls、git status/log）直接放行；medium（write/edit、mutating bash）放行+记 trace；high（黑名单 rm -rf/force-push/publish/curl\|sh/secrets、受保护路径写）🛑 **block+审批**。判定=按工具类型 + 内容扫描（Bash/写入路径）。**总是拦高危，与 `ctx.isProjectTrusted()` 无关**（项目信任只决定能否启动，≠可以 rm -rf）。block 转成错误结果给模型，reason 写明「已被 EvoPi Policy 拦截+原因」。**危险动作黑名单+受保护路径抽成共享 policy 定义**（如 .pi/evopi/policy.json 或代码常量），模块 3 静态扫 skill、模块 4 运行时扫 tool_call 都读它（单一事实源，兑现决策 9）。 |
| 4.3 | 审批 UX（谁批、怎么批） | 🟡 方案已定 | **审批内联在门禁**：high 判定后先 `ctx.ui.confirm("EvoPi 拦截高危", "<命令> 是否放行?")` → 批准=handler 返 undefined 放行+记 `policy.approved`；拒绝=返 `{block,reason}`+记 `policy.denied`；job 状态联动 waitingApproval。**无 UI 模式（hasUI=false）= fail-safe**：弹不出审批就一律 block（reason 说明无法审批）。**审批粒度=支持「本会话都批准这个命令」**，但⚠️收紧作用域：①「记住」只对**具体命令指纹**生效（rm -rf /tmp/x 记住≠所有 rm 放行）②最危险的 blocked 级不允许「记住」，每次都问。理由：无 UI 保守 + 记住便利，但记住必须收紧否则一次误批放行整会话。 |
| 4.4 | checkpoint / rewind | 🟡 方案已定 | **checkpoint=加标签（零成本）**：Pi 每 turn 已有不可变 SessionEntry，checkpoint 只需 `ctx.setLabel(entryId, "checkpoint-before-<tool>")` + 存进 job 的 checkpoint 列表 `{entryId,label,时间,原因}`。**仅危险变更前（medium/high 写操作）自动打**（回退点都是改动前的干净点，少噪声）+ `/evopi-job checkpoint` 手动。**rewind=V1 只做手动** `/evopi-job rewind <checkpoint>`（调 `ctx.navigateTree`）；**自动 rewind 推 V2**（依赖 Eval 判定「何为该回退的失败」，误触发比不回退更糟）。checkpoint 打在危险写前、rewind 回到动手前，语义闭环。 |
| 4.5 | 验收（acceptance 基于证据） | 🟡 方案已定 | **V1=证据面板+辅助判定，人拍板**（自动裁决留 V2 靠 Eval，因「满足验收标准」是语义判断无 Eval 易误判）。`/evopi-job` 汇总 trace 证据：本 job 期间 tool_call/error 数、有无高危被拦、**测试是否通过**；用户切 `passed` 时若证据有 error/高危未解决则⚠️警告。**测试证据=启发式识别测试命令**（npm test/pytest/go test 等）+ 结合 tool_result 的 isError 判过没过。与「V1 观测、V2 自动化」节奏一致。 |
| 4.6 | 写 Trace 的事件结构 | 🟡 方案已定 | 独立 `job.*` / `policy.*` 事件（同 Cost/技能记忆，不污染 Trace core）。Job：已有 start/plan/acceptance/status + 补 `job.checkpoint`/`job.rewind`；Policy：`policy.check`（judgment）/`policy.blocked`/`policy.approved`/`policy.denied`。**套用 3.6 session anchor 判据**：写 session anchor=审批决策（blocked/approved/denied，stub 要求可追溯）+ checkpoint/rewind + job.start/终态 status；只进 JSONL=`policy.check`（逐次风险判定，量大）。**`policy.check` 只记 medium/high，low 不记**（只读无治理价值，不刷 trace）。 |
| 4.7 | 最终方案文档 | ✅ 已落文档 | 已写 [04-执行治理机制.md](04-执行治理机制.md)：含「Pi 现状→EvoPi 增量→实现前后对比」段（首个用新规则的文档，可作前三模块补写样板）+ 8 条决策 + 实现 + 演化路线 V1-V3 + 优点 + 缺点。 |

### 已查实事实（源码确认，全部可行）

- ✅ `tool_call` 返回 `{block:true, reason}` 能**真阻断**（转成错误结果返回给模型，非静默跳过）；`event.input` 可原地改（改参数后放行，不重校验）；多扩展 early-exit（第一个 block 立即返回）。
- ⚠️ Pi **无工具级审批**（只有 project_trust 全局开关）→ EvoPi 自己做；但 `ctx.ui` 有 `confirm/select/input/custom` 交互能力，审批 UX 用 `ctx.ui.confirm()`。
- ✅ session tree 原生 `ctx.fork(entryId)` / `ctx.navigateTree(targetId)` + `ctx.setLabel`；隐式 checkpoint 已有（每 turn 一条 SessionEntry）。**自动 rewind 要自己做**（检测失败→navigateTree）。
- ✅ `input` 钩子能 transform/handled 改写拦截，可把请求包成 job。

## 已锁定的全局决策（跨模块约束）

### 核心模型

- **Session** = 用户与 agent 的主线语义记录（只记重要语义事件）
- **Session Tree** = Session 的 resume / fork / branch 结构
- **Trace** = agent 执行过程的完整观测证据链（记所有执行细节）
- **Eval** = 对 Trace 的评分
- **Memory / Skill** = 从高分/低分 Trace + Eval 沉淀的长期资产

### Trace 关键决策

- 不全量双写 trace 到 session；也不完全断开 session 与 trace。
- 采用 **Anchor-only session writes**：session 只写 traceId / path / summary / checkpoint 等关键锚点。
- 完整 Trace 进 **JSONL**（SQLite 列为后续规划，尚未实现）。
- Trace 是底座；Cost / Memory / Job / Tools / Eval 是上层消费者。
