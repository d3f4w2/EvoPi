## 一句话定位

把整个 EvoPi「产品化」——讲清楚、能演示、可度量、能对外交付。这是**元模块**：管的是 EvoPi 自己怎么被呈现，不涉及 Pi 运行时能力（故本文档无「Pi 现状对比」段，改为「交付物盘点」）。

**项目定位 = 真实可用的开源工具**（面向用户能装能用）。**V1 交付 = 文档体系收口 + 项目总览叙事**；demo 和度量待代码实现后补（因「度量必须能产出才声称」）。

## 二、决策（含理由）

### 决策 1：项目定位 = 真实可用的开源工具

**选择**：EvoPi 面向用户（能装、能用、README 讲得清），而非只是简历作品或学习记录。

**为什么**：定位决定文档产品化的重心。作为「开源工具」，文档要面向使用者——README 是仓库门面、要有「是什么/解决什么/怎么开始」；每模块要有可跑 demo。（若只是简历作品，重心会是叙事亮点；若是学习记录，重心是推理过程。）这个定位也反过来约束前 6 个模块：功能要真能跑、边界要清晰。

### 决策 2：V1 只做文档收口 + 叙事，demo/度量待实现后补

**选择**：V1 交付文档体系收口 + 面向用户 README；每模块的可跑 demo 和量化度量等代码实现后再填。

**为什么**：stub 明确「**度量必须 trace 或测试能产出才声称**」——代码没实现前，任何「缓存命中率提升 X%」「拦截 N 次高危」都是空话。V1 阶段（纯规划）诚实的做法是：把「讲清楚」做扎实，把「演示/度量」留到有真实产出时。硬写纸上 demo 反而违背 stub 自己的规则，也会误导读者以为功能已就绪。

### 决策 3：Demo 用统一模板，度量挂在 trace 事件上

**选择**：每模块 demo 遵循固定模板；成功度量必须对应可观测的 trace 事件/命令输出。

**为什么**：统一模板让 demo 可比较、可复现。度量挂 trace 事件是「可产出才声称」的落地——每个度量都要能指到一个具体的、可验证的信号，而非主观描述。这也让 demo 天然复用 Trace 模块的证据链。

### 决策 4：文档规则作为项目级约定，宿主在 README

**选择**：三条核心文档规则（决策写理由 / 演化路线 / Pi 现状对比）+ 中文书写 + 逐模块讨论，都写在 README 作为项目级约定。

**为什么**：这些规则是讨论过程中逐步确立的（规则 2、3 分别在模块 3、4 时加），需要一个稳定的宿主让后续贡献者对齐。README 是所有人的入口，规则放这里最醒目。规则 3「Pi 现状 → EvoPi 增量 → 实现前后区别」尤其重要——它是 EvoPi「不重复造轮子」哲学的文档保证。

## 三、实现（规范与模板）

### Demo 模板（每模块一个，实现后填）

```text
## <模块名> Demo
- Setup：扩展路径 / 启动命令
- 任务：一个简短的用户任务（如「让 agent 跑 rm -rf /tmp/x」）
- 预期 trace 事件 / 输出：（如 policy.blocked 事件 + 审批弹窗）
- 成功度量：可验证信号（如「policy.blocked 计数 +1，操作未执行」）
- 已知限制：一句话
```

### 度量规范（可产出才声称）

每个模块的成功度量必须能指到具体信号：

| 模块 | 可声称的度量 | 产出来源 |
| --- | --- | --- |
| Trace | 跑一次生成 JSONL 文件 | `.pi/evopi/traces/*.jsonl` 存在 |
| Cost | 缓存命中率、本会话成本 | usage 计算 + `/evopi-cost` |
| 技能记忆 | 记忆条数、候选/审批数 | memory 文件 + `memory.*` 事件 |
| 执行治理 | 高危拦截次数、审批率 | `policy.blocked`/`approved` 计数 |
| 工具运行时 | 错误分类分布、工具延迟 | `tool.result`（errorClass/latency） |
| 评测协作 | golden task 通过率、回归门禁拦截 | `eval.*` 事件 |

### 持续维护机制

- **文档与实现同步**：代码实现后回填 demo/度量，并校准文档描述与真实行为（stub「module docs match implemented behavior」）。
- **进度表是单一事实源**：[进度.md](进度.md) 持续维护，每次工作先看它、完成即更新。
- **外部参考只引不抄**：开源参考项目在调研文档里，借鉴的思想在正文说明，不 copy 代码（README 已声明）。
- **行为级图优于文件清单**：需要图时画行为/数据流，不堆文件目录（且 stub 要求「图只在行为决策完成后加」）。

## 五、架构优点

1. **诚实**：V1 不吹测不出的度量，把「讲清楚」和「演示」分开——有真实产出才声称，符合工具的可信度要求。
2. **文档即产品**：README 主页化让新人打开仓库就懂「EvoPi 是什么、解决什么」，符合开源工具定位。
3. **规则可传承**：文档规则宿主在 README，后续贡献者有稳定约定可对齐；规则 3 保证「不重复造轮子」的透明。
4. **复用证据链**：demo 度量挂在 Trace 事件上，产品化直接吃 Trace 底座的红利，不另造度量。
5. **收尾闭环**：作为元模块，它把前 6 个模块串成一个对外讲得清的整体。

---

## 六、少量缺点

1. **Demo 是空的**：V1 只有模板没有真 demo，「能演示」这个价值要等实现后才兑现——当前对外只能讲设计、不能跑。
2. **度量待验证**：度量规范列了「可声称什么」，但实际数字要实现后跑出来才有，现在是承诺不是事实。
3. **文档易与实现漂移**：一旦进入实现，文档和真实行为可能不同步——需靠「文档与实现同步」维护机制持续对齐，有维护成本。
4. **元模块无独立运行时价值**：它不加任何 agent 能力，纯粹是「关于 EvoPi 的文档」——价值间接，靠让其它 6 个模块更好被理解/使用体现。
5. **叙事随定位变**：若项目定位从「开源工具」转向其它（如纯简历），README 重心要重写——当前主页化是绑定「工具」定位的。
