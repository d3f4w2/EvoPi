# Context Cost Harness 最终方案

> 模块 2 / EvoPi 模块栈。状态：方案已定（2026-06-30），待实现。
> 本文档遵循 [README.md](README.md) 的文档规则：每个决策都写明「为什么这么选」。
> 进度见 [进度.md](进度.md)。

## 一句话定位

Context Cost Harness 是 EvoPi 的**成本观测与缓存优化**模块。它是 Trace 底座之上的第一个消费者，给出快速可量化的产出：缓存命中率、token/成本累计、上下文压力告警。

**V1 范围**：只做「缓存观测 + 预算观测」。**不**裁剪历史、**不**自动 compaction、**不**做 codebase RAG、**不**做 agent router。

---

## 一、决策（含理由）

### 决策 1：缓存这半只做「观测 + retention 策略」，不做主动 cache_control 注入

**选择**：A（命中率观测）+ B（retention 档位显示与建议）；C（主动注入 `cache_control` 断点）推迟到 V2。

**为什么**：查 Pi 源码（[anthropic.ts](../../pi/packages/ai/src/providers/anthropic.ts)）确认，**Pi 已经自动在 Anthropic 请求上打 `cache_control`**，覆盖三个最该缓存的位置：①每个 system text 块 ②最后一个 tool 定义 ③最后一条 user 消息的最后一个内容块。而 Anthropic 总共只允许 **4 个** `cache_control` 断点，Pi 已经占用 3 个。

- 若我们再手动注入断点 → **重复劳动**（这三块已经缓存了），甚至**有害**（可能挤掉 Pi 自己的断点，超过 4 个会报错或被截断，反而降低命中率）。
- 唯一真正有价值的缓存杠杆是 `cacheRetention` 档位（`short` 5 分钟 / `long` 1 小时），这是设一个 Pi 已有的选项，不是注入 payload。

所以 V1 把精力放在「**量化缓存效果**（命中率）+ **用对档位**（retention 建议）」，而不是和 Pi 的自动缓存抢断点。这也正好兑现「主动优化」的定位——retention 是真实可调的杠杆。

### 决策 2：`/evopi-cost` 双命令形态

**选择**：
- `/evopi-cost` —— 摘要快照（上下文压力+档位、缓存命中率、本会话累计 token/成本、请求数）。
- `/evopi-cost detail` —— 原始 usage / payload 摘要 / trace 文件路径。

**为什么**：主视图要清爽好读，把一坨原始 JSON（现状的 `lastPayload` 那行）下沉到 `detail` 子命令。这与现有 trace 命令风格一致（`/evopi-trace last|path`），用户心智负担低。

### 决策 3：上下文压力告警 = 提醒 + 记 trace，绝不自动动上下文

**选择**：上下文压力跨过 **80%(warning) / 90%(high) / 95%(critical)** 时：
- `ctx.ui.notify` 主动弹一次提醒；
- 写一条 `cost.pressure` 到 trace；
- **不**自动 compaction、**不**自动切 retention。

**为什么**：这是全局已锁定的「超阈值只提示+记录，compaction 交给后续模块或用户显式操作」原则的落地。Cost 是观测/建议者，不是上下文内容的裁剪器——自动 compact 会破坏用户对上下文的掌控，且 compaction 语义属于将来的 Memory/Execution 模块。

### 决策 4：新增独立 `cost.*` 事件，不污染 `provider.*`

**选择**：新增三个事件，`provider.request`/`provider.response` 保持原样。

| 事件 | 触发时机 | 关键字段 |
| --- | --- | --- |
| `cost.request` | `before_provider_request` | `contextEstimate{tokens,window,percent}`、`retention`、`messageCount`、`toolCount` |
| `cost.cache` | `after_provider_response` 后读 usage | `usage{input,output,cacheRead,cacheWrite,cacheWrite1h}`、`cacheHitRate`、`cost{input,output,cacheRead,cacheWrite,total}` |
| `cost.pressure` | 跨过 80/90/95% 阈值时一次 | `level`、`percent`、`tokens`、`window` |

**为什么**：`provider.*` 是 Trace core 的事件，归 Trace 模块管。往底座事件里塞 cost 字段会让 **Trace core 依赖 Cost 语义**，违背「Trace 是底座、Cost 是消费者」的边界。独立事件让两者解耦——将来某 provider 没有缓存数据时，`cost.request` 仍能独立工作。`cost.pressure` 只在「首次跨阈值」记一条而非每轮记，避免 trace 噪声。

### 决策 5：`cost.*` 只写 JSONL，不写 session anchor

**选择**：Cost 事件全部进 `.pi/evopi/traces/<traceId>.jsonl`；不调用 `pi.appendEntry` 写 session。

**为什么**：全局约束是 **Anchor-only session writes**——session 只写 traceId/summary/checkpoint 等关键语义锁点。逐轮的 cost 观测不是「关键语义事件」，写进 session 会污染主线语义记录。

### 决策 6：V1 只 JSONL + in-memory，不建独立 SQLite

**选择**：in-memory 累加器供 `/evopi-cost` 即时读；`cost.*` 落 JSONL；不建 Cost 专属 SQLite。

**为什么**：① 与 Trace 一致——Trace 自己都还没上 SQLite（「等事件词表稳定再加」），消费者不该抢跑。② 够用——V1 只需「本会话快照 + 跨阈值告警」，in-memory + 当轮 JSONL 足矣，`/evopi-cost` 不需要跨会话历史查询。③ YAGNI——SQLite 的价值是跨会话聚合/趋势（如「过去 7 天平均命中率」），那是 dashboard 级需求。将来 Trace 上 SQLite 时，结构化的 `cost.*` 可直接复用同一 store，无需自立门户。

## 二、实现

### 钩子与数据来源（均已在 Pi 源码确认）

| 用途 | API | 备注 |
| --- | --- | --- |
| 请求侧观测 | `pi.on("before_provider_request", ...)` | `event.payload` 是裸 Anthropic 请求体 |
| 响应侧触发 | `pi.on("after_provider_response", ...)` | 仅 `{status, headers}`，**不含 usage** |
| 上下文用量 | `ctx.getContextUsage()` | `{tokens,contextWindow,percent}`，**tokens 是估算**，compaction 后到下次响应前为 null |
| 真实 token/成本 | `ctx.sessionManager.getLeafEntry()` 的 assistant message `.usage` | `Usage` 含 `cacheRead/cacheWrite/cacheWrite1h` 及**已算好的** `cost{...}`（美元），直接取用 |
| 命令 | `pi.registerCommand("evopi-cost", {...})` | 现有 MVP 已注册，扩展之 |
| 告警 | `ctx.ui.notify(text, "warning")` | |

### 处理流程

1. **`before_provider_request`**：记 `cost.request`（contextEstimate / retention / msg+tool count）；同时检查 `getContextUsage().percent`，若首次跨过某阈值 → `ctx.ui.notify` + 记 `cost.pressure`。
2. **`after_provider_response`**：读 `leafEntry.usage`，算 `cacheHitRate = cacheRead / (cacheRead + input)`，记 `cost.cache`（含 usage 与 cost）；in-memory 累加 token 与 cost。
3. **`/evopi-cost`**：渲染 in-memory 累加器 + 当前 `getContextUsage()` + retention 档位 → 摘要快照。
4. **`/evopi-cost detail`**：额外展示最近 payload 摘要与 JSONL 路径。

### 缓存命中率口径

`cacheHitRate = cacheRead / (cacheRead + input)`。`input` 为本轮未命中缓存的输入 token；`cacheRead` 为命中缓存读取的 token。值越高代表稳定前缀复用越好。

### 实现落点

复用现有扩展文件 `.pi/extensions/evopi-trace/index.ts`（`/evopi-cost` 与 provider 钩子已在其中）。本模块是对它的扩展，不新开文件——与「Cost 暂不并进 Trace core，但物理上先共存于同一扩展」的现实一致；将来模块拆分时再迁出到 `evopi-cost/`。

## 三、架构优点

1. **快速可量化**：缓存命中率、token、成本都是**真实数据**（来自 provider usage，非估算），是 EvoPi 第一个能直接展示「省了多少钱」的模块，适合做 demo 与简历叙事。
2. **零侵入、不打架**：不绕过 Pi provider 适配层，不和 Pi 自动缓存抢断点；用的全是 Pi 官方机制（`cacheRetention`）。风险面极小。
3. **边界干净**：`cost.*` 独立事件 + 只写 JSONL + 不建独立 SQLite，严格遵守「Trace 是底座、Cost 是消费者」与「Anchor-only」两条全局约束，不给后续模块挖坑。
4. **演进友好**：结构化 JSONL 事件将来可直接导入 Trace 的 SQLite / OTLP / Langfuse；C 层（主动注入）与跨会话聚合都留好了升级位。

---

## 四、少量缺点

1. **`getContextUsage().tokens` 是估算**，与 provider 真实计数有偏差；compaction 后到下次响应前为 null，此时压力百分比会短暂不可用（需在 UI 上标注「估算」并对 null 兜底）。
2. **缓存命中率有滞后**：必须等 `after_provider_response` 后读 usage 才能算，无法在请求发出前预测命中率。
3. **retention 切换是「建议不自动」**：V1 只提示用户切 `long`，不自动切——长会话可能因为用户没切而错过 1h 缓存收益（保守换安全，自动切 retention 留到 V2 且需先确认扩展能否动态改）。
4. **跨会话视角缺失**：无 SQLite 意味着 `/evopi-cost` 只看当前会话，看不到历史趋势（这是 V1 的有意取舍）。
5. **物理上仍寄居 trace 扩展**：Cost 逻辑暂与 Trace 共存于同一文件，模块边界目前靠约定而非目录隔离，等后续拆分才物理分离。
