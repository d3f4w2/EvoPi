# 03 技能记忆机制（Skill Memory Harness）最终方案

> 模块 3 / EvoPi 模块栈。状态：方案已定（2026-06-30），待实现。
> 遵循 [README.md](README.md) 文档规则：每个决策写明理由；给出演化路线。
> 进度见 [进度.md](进度.md)。

## 一句话定位

把核心模型里「Memory/Skill = 从高分/低分 Trace + Eval 沉淀的长期资产」落地。管理项目记忆 + 大规模技能集，不污染 prompt，并支持受控的自我进化。

**V1 范围 = A（记忆）+ B（技能路由/治理）**。C（技能自进化、棘轮）推到 V2，因为它依赖还没做的模块 6 Eval，且「agent 改自己」安全面大。

**EvoPi 在 Pi 之上补的缺口**（Pi 原生 skills 是纯静态加载，无下列能力）：Top-K 路由、trust 分级、使用统计、从 Trace 自动生成候选、项目记忆。

---

## 一、决策（含理由）

### 决策 1：Memory 与 Skill 双层分开，同一模块管

**选择**：两层独立（存储/路由/生命周期各自独立），但都在模块 3 内。

**为什么**：Memory 是「数据」（被读），Skill 是「可执行逻辑」（被调用），三点根本不同——① **信任级别**：记忆错了顶多误导，技能错了会执行危险操作（含 `rm`/`git reset`），混在一起信任模型说不清；② **注入策略**：记忆短、直接给全文，技能可能长、必须 Top-K + 摘要 + 按需展开；③ **演化路径**：记忆是「积累与遗忘」，技能是「生成与棘轮淘汰」。现有 MVP 已天然分 `MEMORY.md` / `SKILLS.md` 两个文件，本决策只是把划分正式化。

| 维度 | Memory（记忆层） | Skill（技能层） |
| --- | --- | --- |
| 本质 | 项目事实/偏好/决策/约束 | 可复用工作流/操作步骤 |
| 进上下文 | 检索后整条注入 | 只注入 name+description，全文按需 |
| 信任 | 低风险 | 需 trust 分级 |
| 生命周期 | 可编辑/删除，带 scope | 候选→审批→启用→统计→（棘轮）保留/回滚 |

### 决策 2：记忆分 project / user / global 三级 scope

**选择**：三级；冲突时 **project > user > global**（就近原则）。

| scope | 存储位置 | 含什么 |
| --- | --- | --- |
| project | `.pi/evopi/memory/`（随仓库可 commit） | 本项目特有事实/决策/约束 |
| user | `~/.evopi/memory/`（跨项目） | 个人偏好 |
| global | 内置/团队共享（V1 留接口） | 普适规则 |

**为什么**：项目级约束应覆盖个人习惯，个人习惯覆盖普适默认——与 Pi/Claude Code 自己的 settings 层叠模型一致。`/evopi-memory add` 默认写 project，`--user`/`--global` 写对应层；每条记忆带 `scope` 字段。

### 决策 3：技能层扩展 Pi 的 SKILL.md，不另起炉灶

**选择**：复用 Pi 的 `SKILL.md` 加载 + 注入机制，把 EvoPi 治理字段（trust/统计/source-trace-id/审批状态）塞进 frontmatter 额外键。

**为什么**：查源码确认 Pi frontmatter 是 `[key: string]: unknown`——额外字段会被**保留但不解释**，且 Pi 明确「provenance 由应用层定义」。所以 EvoPi 的治理字段可以寄生在 Pi schema 里，Pi 照常加载，EvoPi 自己读这些字段做治理。这与 Cost「能复用 Pi 机制就不另起炉灶」同一哲学：不重复造加载/注入轮子，自动吃 Pi 升级红利，一份文件两层都用。

### 决策 4：Top-K 路由的挂点——Skill 走 resources_discover，Memory 走 context（已查证修正）

**选择**：**Skill** 用 `resources_discover` 钩子按 trust 过滤；**Memory** 用 `context` 钩子改 messages 注入。

**为什么**（这是一次被源码事实修正的决策）：原本设想「在 `context` 钩子做 skill Top-K」，但查证发现——
- Pi 系统提示里**只放 skill 的 name+description 列表**（`formatSkillsForPrompt`），完整内容不进上下文，只在 `/skill:name` 显式调用或模型读该文件时加载。**Pi 自己已经做了渐进披露**。
- `context` 钩子**碰不到 skill**：系统提示在 context 钩子触发**之前**就固定了（`ContextEvent` 只有 messages、无 systemPrompt）。
- 正确的 skill 挂点是 `resources_discover`（控制哪些 skill 文件被加载）或 `before_agent_start`（改 systemPrompt）。
- 而 Memory 注入走 messages，`context` 钩子可行。

→ 两层挂点不同，必须分开实现。

### 决策 5：V1 的 Skill 路由只做 trust 过滤，不做 Top-K 数量裁剪

**选择**：`resources_discover` 里按 trust/审批状态过滤（未审批候选不加载进上下文）；数量裁剪留 V2。

**为什么**：Pi 已做渐进披露——50 个 skill 也只是 50 行 name+description，不是 50 篇全文。所以「不污染 prompt」V1 已由 Pi 达成，EvoPi 的数量裁剪只在 skill 多到「连列表都太长」时才有价值。**V1 最大的真实风险是低可信 skill 被模型看见**（用户可能 `npx skills add` 引入第三方），所以 V1 聚焦 trust 过滤这个治理需求，而非 token 需求。

### 决策 6：Memory 检索用关键词 + 时间邻近（本地零 API）

**选择**：本地打分——query 词频 + tags 命中 + 最近优先，取 Top-K 注入；不用 LLM 排序、不全量注入。

**为什么**：mempalace 证明纯本地方案（零 LLM/零 API）能到 LongMemEval R@5 96.6%，对项目记忆这种规模完全够；零成本、零延迟、可解释，与 Cost「零 API 优先」一致。全量注入会随记忆增长爆 context（与「不污染 prompt」矛盾）；LLM 排序每轮多一次调用，与省钱目标冲突。语义检索留 V2。

### 决策 7：记忆写入——保守生效 + 提前积累

**选择**：
- `/evopi-memory add` 显式写 → **直接生效**（用户负责）。
- `session_before_compact` 压缩前自动抢救 → 写成**候选**（`status: candidate`），不自动注入，等 `/evopi-memory review` 确认。
- `message_end`/`tool_result` 每轮自动抽取 → **V1 不做**（留 V2）。
- 技能统计：V1 通过 `tool_call`/`tool_result` **采集但不用**（记调用次数/成败），为 V2 棘轮铺数据。

**为什么**：记忆会被注入回未来上下文，**错误/噪声记忆会污染所有后续会话**。每轮自动抽取会攒一堆「用户问了 X」的噪声，且要额外 LLM 判断「值不值得记」，收益不明。`session_before_compact` 是 claude-mem/mempalace 的核心触发点，能解决「失忆」痛点，但写成候选过人审才安全——与技能「自动产生的都是候选」同一治理哲学。技能统计现在采集，等模块 6 Eval 就绪后 V2 棘轮才有历史数据可用（与 Cost「先观测后优化」同节奏）。

### 决策 8：Skill trust 四级状态机 + 审批闸门（V2 棘轮的地基）

**选择**：trust 写 SKILL.md frontmatter，四级：

| 级别 | 含义 | resources_discover 行为 |
| --- | --- | --- |
| `trusted` | 人工写/已审批 | ✅ 加载，模型可见可调 |
| `candidate` | 自动生成或第三方新引入，待审 | ⚠️ 不加载进上下文，可 `/evopi-skill review` 查看 |
| `untrusted` | 来源不明/含可疑动作 | ❌ 不加载 |
| `blocked` | 审查命中危险动作，已封禁 | ❌ 永不加载 |

来源定初始 trust：人工写→`trusted`，第三方/自动生成→`candidate`，命中危险黑名单→`blocked`。命令：`/evopi-skill review`（列 candidate + 来源 trace + 危险扫描结果）、`approve`、`block`。候选元数据带 `source`/`evidence`/`approved_by`/`rollback`。

**为什么 V1 就做 trust（哪怕自进化在 V2）**：trust 是 V2 棘轮的**前置安全闸**——没有它，V2 自动生成的 skill 会直接可调用，这正是「agent 改自己」最危险处。先建好闸门，V2 才敢往里灌自动生成的 skill。且第三方 skill 的信任问题 V1 就存在。

### 决策 9：危险动作扫描 = 黑名单 + 工具/路径解析（复用模块 4/5 定义）

**选择**：扫 SKILL.md 内容里的黑名单词（`rm -rf`/`git reset --hard`/force-push/`curl|sh` 等）+ 解析 skill 声明要调用的工具/受保护路径；命中则 review 标红 + 建议降 `blocked`。

**为什么 + 协调点**：这是 darwin-skill「高危动作黑名单」的落地。⚠️ **与 Execution（模块 4）/Tool Runtime（模块 5）的策略门禁有职责重叠，必须协调**——区别是：模块 3 扫描 = **静态**审查 skill **文本**「声明要用什么」→ 决定是否准入可见；模块 4/5 门禁 = **运行时**拦截**实际**危险 `tool_call` → 决定是否放行执行。两者是两道防线（静态准入 + 运行时拦截），不冲突。但模块 3 的工具/路径解析器应**复用**模块 4/5 的危险动作定义，不各搞一套黑名单。

### 决策 10：独立 `memory.*` / `skill.*` 事件，session anchor 看「资产产生 vs 使用」

**选择**：新增独立事件，不污染 `provider.*`/Trace core（同 Cost 2.2）。

| 侧 | 事件 |
| --- | --- |
| Memory | `memory.write`、`memory.review`、`memory.retrieve` |
| Skill | `skill.invoke`、`skill.outcome`、`skill.approved`、`skill.blocked`、`skill.filtered` |

**session anchor 判据**（细化 Anchor-only，可复用到后续模块）：**资产的产生/审批 = 语义锁点 → 写 session anchor**（`skill.approved` + 显式 `memory.write`）；**资产的逐次使用/观测 = 执行细节 → 只进 JSONL**（`skill.invoke`/`outcome`/`memory.retrieve`）。

**为什么**：Cost 全是观测，故 `cost.*` 一律只进 JSONL；但 Memory/Skill 有「资产变更」这种真正的语义事件，比 Cost 更该有 session anchor。这与接力上下文早列的 `evopi.skill.accepted` 应入 session 一致。

## 二、实现

### 钩子与挂点（均已源码确认）

| 用途 | 挂点 | 备注 |
| --- | --- | --- |
| Skill 按 trust 过滤加载 | `resources_discover`（返回 skillPaths） | `context` 钩子碰不到 skill |
| Memory 检索注入 | `context`（改 messages） | 注入 Top-K 记忆条 |
| 记忆压缩前抢救 | `session_before_compact` | 写成 candidate |
| 技能统计采集 | `tool_call` / `tool_result` | 记调用/成败 |
| 命令 | `registerCommand` | `/evopi-memory`、`/evopi-skill` 系列 |

### 存储布局

```text
.pi/evopi/memory/                 # project scope（user 在 ~/.evopi/memory/ 同构）
  INDEX.md                        # 分类地图 + token 成本（claude-mem 渐进披露）
  MEMORY.md                       # 记忆条目（结构化小节，非流水账）
  SKILLS.md                       # 技能候选索引
.pi/evopi/skills/<name>/SKILL.md  # 技能本体（Pi 格式 + EvoPi 治理 frontmatter）
```

### 记忆条目格式（轻量元数据 + Markdown 正文）

```markdown
## [project] commit 必须用中文
- scope: project
- source: trace tr_xxx / 用户显式
- created: 2026-06-30
- status: active        # 或 candidate
- tags: [git, 规范]

EvoPi 所有 commit message 用中文，小步提交。
```

可 diff / 可审查 / 可检索（按 tags+关键词）/ 可手写。

### SKILL.md 治理 frontmatter（示例）

```yaml
---
name: publish-npm
description: 发布 npm 包的标准流程
trust: candidate
source: trace tr_xxx       # V2 自动生成时
evidence: eval_xxx         # V2
approved_by: null
rollback: git
---
```

### 命令集

- `/evopi-memory`、`/evopi-memory add <fact> [--user|--global]`、`/evopi-memory review`
- `/evopi-skill`（列表+trust）、`/evopi-skill review`、`/evopi-skill approve <name>`、`/evopi-skill block <name>`

### 实现落点

V1 复用 `.pi/extensions/evopi-trace/index.ts`（`/evopi-memory` MVP 已在其中）。模块拆分后迁到 `.pi/extensions/evopi-memory/`。

## 三、演化路线

| 版本 | 范围 | 说明 |
| --- | --- | --- |
| **V1（当前）** | A 记忆 + B 技能治理 | 三级 scope 记忆、关键词检索注入、压缩前抢救成候选、显式 add；Skill 扩展 SKILL.md、trust 四级闸门、危险扫描、统计采集（不用） |
| **V2** | C 技能自进化（棘轮）+ 检索增强 | 依赖模块 6 Eval：从高分 Trace 自动生成候选 skill → 多评估器打分 → 棘轮保留/回滚（基线锁最佳、<1 分早停、git-revert 兜底）；记忆 `message_end` 自动抽取 + 类型化（profile/event/knowledge/behavior/skill/tool，memU 风格）；Skill Top-K 数量裁剪；记忆语义检索 |
| **V3** | 跨项目资产 + 自治 | user/global 记忆跨项目共享与同步；skill 跨项目分发；自动遗忘/衰减；记忆冲突自动消解 |

## 四、架构优点

1. **复用 Pi、缺口精准**：不造加载/注入轮子，只补 Pi 缺的治理层（trust/路由/统计/记忆）；与 Cost 同一「不另起炉灶」哲学，风险面小。
2. **治理哲学统一**：Memory 候选、Skill 候选、自动抽取——「自动产生的都是候选，要过人审」贯穿始终，与 darwin-skill「人在环」一致，避免「agent 改自己」失控。
3. **可读可审查可溯源**：记忆是带元数据的 Markdown（可 diff、可单独增删），每条带 source 溯源；trust 写在 SKILL.md 自包含；延续 EvoPi「证据链」哲学。
4. **为 V2 铺好地基**：trust 闸门 + 技能统计采集在 V1 就位，V2 棘轮自进化可直接踩在上面，无需返工。
5. **零成本检索**：本地关键词+时间邻近，不增加每轮 token/延迟。

---

## 五、少量缺点

1. **治理字段寄生在 Pi schema**：trust/source 等是 frontmatter 约定键，非强类型——Pi 若未来收紧 frontmatter 校验可能受影响（概率低，Pi 明确支持额外键）。
2. **关键词检索召回有上限**：同义词/语义相近的记忆可能漏召（如查「省钱」命不中标了「成本」的记忆），需 V2 语义检索补足。
3. **候选审批靠人**：压缩抢救的记忆、第三方 skill 都要人工 review，记忆/技能多时审批有负担（换来的是不被噪声污染）。
4. **危险扫描与模块 4/5 有重叠**：静态扫描的工具/路径定义需与运行时门禁协调复用，否则两套黑名单易不一致——这是个需在实现期主动对齐的协调点。
5. **统计采集但不用**：V1 采集 skill 成败却不据此决策，短期看是「只记不用」的开销，价值要等 V2 才兑现。
6. **user/global scope V1 仅占位**：三级 scope 定了，但 user/global 的跨项目存储与同步 V1 只留接口，真正跨项目能力是 V3。
