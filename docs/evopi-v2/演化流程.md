# EvoPi 演化流程（新输入怎么变成改动，而不把项目搅乱）

> V1 用 [`impl-harness`](../evopi-v1/impl/README.md) 解决了「方案已定 → 有序实现」。
> 这份文档解决下一个问题：**项目已经跑起来了，新输入不断进来（读了文章、看了开源项目、自己冒出灵感、或觉得某处架构不优雅），怎么让它们有序地改造项目，而不是一有想法就乱。**
>
> 一句话原则：**捕获与评判分离，决策留痕，契约可解冻但要走流程。**

---

## 为什么需要它

个人项目最容易死在两种极端：

1. **来者不拒**：读一篇文章就想重构一次，计划天天变，什么都做不完。
2. **来者不理**：埋头做既定计划，好点子当场拒掉或忘掉，项目停止进化。

这份流程是中间道：**任何新输入都先无门槛地"接住"，再在固定的节奏里冷静地"消化"**——够格的才动手，不够格的留痕存档（避免下次又捡起来重新纠结）。

---

## 五步漏斗

```text
新输入（文章 / 开源 repo / 自己的灵感 / "这里不优雅"）
   │
   │  ①  捕获 —— 一行字先记进 灵感池.md，不评判、不打断手头的事
   ▼
   ②  提炼 —— 有空时回看：它到底主张什么？对 EvoPi 哪个模块有可借鉴点？
   │           （参考 V1 已有的调研范式：docs/Pi开源参考项目调研_2026-06-30.md）
   ▼
   ③  定性 —— 归到四类之一：
   │      ├─ A 新功能        → 进 V2路线图.md 排优先级
   │      ├─ B 架构优化      → 写一篇 ADR（架构决策记录），评估「爆炸半径」
   │      ├─ C 推翻旧决策    → 写 ADR + 更新 V1 的冻结契约/变更日志（记清"为什么改"）
   │      └─ D 不适合        → 灵感池里标 ✗ + 一句理由（"看过，不用，因为___"）
   ▼
   ④  排期 —— 够格的进路线图，标优先级（P0/P1/P2）；不够格的留在灵感池
   ▼
   ⑤  实施 —— 仍走 V1 那套：冻结契约 → 拆文件接线 → 核心逻辑 → 自测 → 小步中文提交
```

**每一步的产物落点**：

| 步 | 动作 | 落在哪 | 花多久 |
| --- | --- | --- | --- |
| ① 捕获 | 记一行 | [灵感池.md](灵感池.md) | 10 秒 |
| ② 提炼 | 写两三句"它主张什么/可借鉴点" | 灵感池.md 同一条下 | 几分钟 |
| ③ 定性 | 打 A/B/C/D 标签 | 灵感池.md | 1 分钟 |
| ④ 排期 | A/B/C 类升级到路线图或 ADR | [V2路线图.md](V2路线图.md) / [架构决策记录/](架构决策记录/) | 几分钟 |
| ⑤ 实施 | 编码 | 代码 + 提交 | 按模块 |

## 三条要害规则（让它"不乱"的关键）

### 规则 1：捕获 ≠ 评判（时间上分开）

灵感来了**只记不判**。当场决定"做不做"有两个坏处：要么打断你正在做的事，要么在没想清楚时草率拒掉一个好点子。捕获是 10 秒的事，评判留到你有整块时间时集中做。

### 规则 2：决策必须留痕（ADR）

任何"要不要改架构 / 要不要推翻一个旧决定"，都写一篇简短的**架构决策记录（ADR）**：背景、备选方案、决定、理由。
- **为什么**：三个月后你自己会忘"当初为什么这么设计"；面试官翻到 ADR 会看到的不是"你会写代码"，而是"**你会演化一个系统、会权衡**"——这是比功能更值钱的信号。
- V1 的[变更日志](../evopi-v1/impl/进度.md#变更日志改冻结约定必须记这里含理由)已经是 ADR 的雏形，ADR 是它的升级版（一决策一篇，结构化）。

### 规则 3：契约可解冻，但要走流程

V1 冻结了一批全局约定（事件词表、Anchor-only、安全>资源、共享 policy、保守默认——见 [impl/进度.md](../evopi-v1/impl/进度.md)）。V2 **不是不能改**这些，而是：

- 改动前先问：这是**加法**（加字段/加模块，向后兼容）还是**改造**（改名/删字段/改结构，会牵动下游）？
- **加法**：低风险，直接做，进度表加一行。
- **改造**：先写 ADR 评估爆炸半径（谁依赖它？SQLite/看板/其它模块会不会崩？），再动。

---

## "爆炸半径"怎么估（改造类专用）

改一处前，问自己三个问题，答案越多"是"越危险、越该先写 ADR：

1. **有没有下游依赖它的稳定形态？**（比如事件字段被 JSONL/SQLite/未来自动化依赖）
2. **动它要不要同时改多个模块？**（比如动 Trace 底座 → 五个消费者全受影响）
3. **它是不是一条已冻结的全局约定？**（动"安全>资源""Anchor-only"这种要格外慎重）

> 经验法则：**加法优先**。能用"加一个新东西 + 逐步迁移"达成的，就不要"原地改结构"。V1 就是这么做的（policy.ts 提前建、单 tool_call handler 收敛都留了变更日志）。

## 和 V1 资产的关系（不另起炉灶）

这套流程刻意**长在你已有的东西上**，不是新发明：

- **提炼范式** → 复用你 V1 已经做过的 [开源参考项目调研](../Pi开源参考项目调研_2026-06-30.md)（4 个 repo 的精华提炼 + 融合建议，就是"从外部输入提炼进项目"的活样板）。
- **实施流程** → 复用 V1 的 [impl-harness 流水线](../evopi-v1/impl/README.md)（冻结契约 → 拆文件 → 小步提交）。
- **决策留痕** → 升级 V1 的[变更日志](../evopi-v1/impl/进度.md)为结构化 ADR。
- **冻结约定** → 继承 V1 的五条全局约定，V2 只在走流程后显式修改。

## 一句话 SOP（贴在心里）

> **想法来了 → 记进[灵感池](灵感池.md)（别停下手头的事）。有空了 → 提炼、定性（A新功能/B架构/C推翻/D不用）。够格的 → 进[路线图](V2路线图.md)或写[ADR](架构决策记录/)。动手时 → 走 V1 老流程。改架构/推翻旧决定 → 先写 ADR 估爆炸半径。**
