# EvoPi V2 · 演化机制

> V1 解决了「方案已定 → 有序实现」（[impl-harness](../evopi-v1/impl/README.md)）。
> V2 解决下一个问题：**项目已经跑起来后，新输入不断进来（文章 / 开源项目 / 灵感 / "这里不优雅"），怎么让它们有序改造项目，而不把计划搅乱。**
>
> 核心不是"V2 要做什么功能"，而是"**怎么持续演化而不乱**"——功能只是演化的产物。

## 四份文件，各干一件事

| 文件 | 角色 | 什么时候看 |
| --- | --- | --- |
| [演化流程.md](演化流程.md) | **SOP（骨架）** —— 五步漏斗 + 三条要害规则 | 想知道"新东西进来该怎么走" |
| [灵感池.md](灵感池.md) | **捕获层** —— 新输入先记这里，不评判 | 任何时候有想法，10 秒记一行 |

| [架构决策记录/](架构决策记录/) | **决策留痕（ADR）** —— 一决策一篇 | 要改架构 / 推翻旧决定时 |

## 新想法进来，走这条路（一句话）

> 记进[灵感池](灵感池.md)（别停下手头的事）→ 有空提炼定性（A新功能 / B架构 / C推翻 / D不用）→ 够格的进[路线图](V2路线图.md)或写[ADR](架构决策记录/)→ 动手走 V1 老流程 → 改架构先估爆炸半径。

完整版见 [演化流程.md](演化流程.md)。

## 它怎么长在 V1 上（不另起炉灶）

- **提炼范式** 复用 V1 的 [开源参考调研](../Pi开源参考项目调研_2026-06-30.md)（4 个 repo 精华 + 融合建议，本身就是"外部输入→项目"的样板）。
- **实施流程** 复用 V1 的 [impl-harness](../evopi-v1/impl/README.md)。
- **决策留痕** 升级 V1 的[变更日志](../evopi-v1/impl/进度.md)为结构化 ADR。
- **冻结约定** 继承 V1 五条全局约定，V2 只在走流程后显式修改（首例见 [ADR-0001](架构决策记录/ADR-0001-v2是否解冻自动决策约定.md)）。
