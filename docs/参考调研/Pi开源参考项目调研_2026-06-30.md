## 1. 记忆类（落地 Skill Memory Harness 的"记忆"部分）

> **共同落点（来自 [基础版本PI.md](基础版本PI.md) 第 4 章）**：
> - 读取/注入 → `context` 挂点（每次调大模型前改 messages）或 `before_agent_start`
> - 启动加载 → `session_start`
> - 捕获新知识 → `message_end` / `tool_result`
> - 压缩前沉淀 → `session_before_compact`
> - 落盘持久化 → `pi.appendEntry`（可跨会话/跨分支恢复）
>
> 三个记忆项目恰好示范了**三种不同的记忆工程路线**，互补而非重复。

### 1.1 ⭐⭐⭐ thedotmack/claude-mem —— 钩子生命周期 + 压缩 + 渐进式披露

- **亮点**
  - **5 个生命周期钩子**（SessionStart / UserPromptSubmit / PostToolUse / Stop / SessionEnd）把记忆直接焊进会话循环——和 Pi 的事件模型几乎一一对应。
  - **渐进式披露（Progressive Disclosure）**：分层检索 `search → timeline → get_observations`，**每一层都显式标注 token 成本**，模型按需下钻，号称省 ~10× token。
  - **Endless Mode**（beta）：面向超长会话的"仿生记忆架构"。
  - `<private>` 标签排除敏感内容；自带实时 Web 查看器（端口 37777）。
- **核心机制**：SQLite（sessions/observations/summaries + FTS5 全文检索）+ Chroma 向量库做语义/关键词混合检索；Bun 管理的 worker 暴露 HTTP API；管线为"捕获工具使用观察 → AI 生成语义摘要 → 带 embedding 存储 → 三层检索"。
- **与 Claude Code 集成**：作为 plugin 装入，通过 5 钩子注入，暴露 4 个 MCP 工具 + `mem-search` skill。
- **成熟度**：Apache-2.0 / TypeScript / v13.4.0 / Node ≥20。
- **借鉴价值（最强的架构模板）**：钩子→事件映射近乎直接：SessionStart→Pi `session_start`（检索+注入）、PostToolUse→`message_end`/`tool_result`（捕获观察）、Stop/SessionEnd→`session_before_compact`（刷新+摘要，落盘走 `pi.appendEntry`）。**优先移植两点**：① 带 token 成本标注的**渐进式披露**，让模型自己决定加载多深；② 在压缩事件上做"观察→语义摘要"的**压缩步骤**，原始观察留 SQLite/FTS，摘要作为廉价默认层。
- **放在 Pi 的部位**：`.pi/extensions/memory/` 一个扩展，订阅 `session_start`+`message_end`+`session_before_compact`，内部持久化用 `appendEntry`。

### 1.2 ⭐⭐⭐ MemPalace/mempalace —— 本地优先、零 API、原文存储

- **亮点**
  - **96.6% R@5（LongMemEval 原始分），全程零 LLM 调用、零 API**；"Hybrid v4"靠关键词加权 + 时间邻近加权 + 偏好模式提取冲到 98.4%。
  - **原文（verbatim）存储**：不做有损摘要，检索时返回精确历史原文。
  - **带有效期窗口的时序知识图谱**（SQLite 支撑）——事实/偏好带时间范围，天然处理"会变的事实"。
  - 每个专精子代理独立"侧厅 + 日记"，做到 per-agent 隔离。本地 embedding 模型（~300MB），数据不出机器。
- **核心机制**：空间隐喻——人物/项目=wings，主题=rooms，原文=drawers；默认 ChromaDB，备选 sqlite_exact / Qdrant / pgvector，用本地 marker 文件防连错库。
- **与 Claude Code 集成**：**35 个 MCP 工具** + **auto-save 钩子**（周期性保存 + **上下文压缩前保存**）。
- **成熟度**：MIT / Python 3.9+ / v3.5.0。
- **借鉴价值（最佳"本地优先、无 API"检索引擎）**：它的 **save-before-compaction 钩子正是 Pi 的 `session_before_compact`**——在原文丢失前 `appendEntry` 落盘的标准触发点。**移植三点**：① **原文优先存储 + 检索时排序**（无损、比压缩管线更易实现）；② **时序有效期图谱**处理长项目中变化的事实/偏好；③ **混合排序加权**（关键词+时间邻近+偏好模式）叠在朴素向量召回上——一个**不需要在 `context` 钩子里调 LLM** 的高 ROI 精度提升。
- **放在 Pi 的部位**：与 1.1 同一个 memory 扩展，但**检索引擎可换成 mempalace 的本地零 API 方案**（更省钱、更可控），`session_before_compact` 是核心触发点。

### 1.3 ⭐⭐ NevaMind-AI/memU —— 可读 Markdown 记忆树 + 自进化技能

- **亮点**
  - **三层 Markdown 文件模型**：INDEX.md（分类地图）→ MEMORY.md（事实/偏好/目标/事件）→ SKILL.md（从执行轨迹自动抽取的工具模式）。记忆**可读、可 diff、可人工审查**，不是黑盒向量。
  - **自进化技能**：SKILL.md 条目从工具轨迹自动抽取，且**每次 `memorize()` 都增量精炼**。
  - **自组织文件夹**：自动建分类、链接、摘要、embedding，无需手工打标。
  - **类型化记忆抽取**：profile / event / knowledge / behavior / skill / tool；Locomo ~92%。
- **核心机制**：Markdown 优先（非向量 blob），可插拔后端（内存/SQLite/Postgres+pgvector），检索可走 RAG 或纯 LLM 排序召回。`memorize()` 写 / `retrieve()` 读两个异步 API。
- **与 Claude Code 集成**：**无** MCP/钩子，需自己包一层（programmatic API + 托管 Cloud REST）。
- **成熟度**：Apache-2.0 / Python 3.13+ / PyPI `memu-py`。
- **借鉴价值（最"可读"的持久化模型）**：**人类可读 Markdown 记忆树完美对齐 `pi.appendEntry`**——持久化 INDEX/MEMORY/SKILL 式 Markdown 而非不透明向量。**移植两点**：① **类型化记忆条目**（profile/event/knowledge/behavior）来结构化 `message_end` 抽取的内容；② **自进化 SKILL.md**——每次会话从事件流蒸馏重复工具模式成可复用技能笔记。其 `memorize`/`retrieve` 切分是 Pi 扩展"写（compact）vs 读（context）"两条路径的干净 API 形状。
- **放在 Pi 的部位**：作为 memory 扩展的**持久化格式参考**（Markdown 树），并与 Skill Memory Harness 的"PI.md 项目记忆"和"自进化 Skill"两块呼应。

> **记忆三选一/组合建议**：
> - 想要**最完整架构模板** → claude-mem（钩子+压缩+渐进披露）
> - 想要**本地零 API、最省钱、精度高** → mempalace（原文+混合排序+时序图谱）
> - 想要**可读可审查的记忆 + 自进化技能** → memU（Markdown 树）
> 推荐：**以 claude-mem 的钩子骨架 + mempalace 的本地检索引擎 + memU 的 Markdown 可读持久化** 三者融合，落成 Pi 的一个 memory 扩展。

## 2. 沙箱 / 运行时隔离类（落地 Tool Runtime Harness 的"沙箱"部分）

> **共同落点（来自 [基础版本PI.md](基础版本PI.md) 第 4.3 节 C 组）**：
> Pi **官方不内置权限系统**，但 `tool_call` 挂点可 **`{block:true}` 拦截** 或 **原地改写入参**（底层落在 agent-loop，拦得住、不是只提示）；配合 `project_trust`、`user_bash`、`ctx.ui.confirm()`。
> 两个沙箱项目代表两条路：**学设计自己用 bwrap 轻量实现** vs **直接把 agent 关进现成的重型沙箱**。

### 2.1 ⭐⭐⭐ opensandbox-group/OpenSandbox —— 防御级隔离 + 出网管控（设计参考首选）

- **亮点**
  - **可选硬化运行时**：gVisor / Kata Containers / Firecracker microVM，**按工作负载逐个选择**，内核级隔离（不止 namespace）。
  - **每沙箱出网/网络管控** + 入口网关——真正的网络沙箱。
  - 通过 **"Sandbox Protocol"** 抽象可插拔运行时（"本地 bwrap"与"远程 microVM"可互换）。
  - **凭据金库（Credential Vault）** 安全注入密钥。
- **核心机制**：FastAPI 控制平面，调度 Docker（本地）/ Kubernetes（分布式）后端；安全平面含统一入口网关 + 出网控制 + 凭据金库。
- **接口**：REST API + CLI（`osb`）+ MCP server + SDK（Python/Java-Kotlin/JS-TS/C#-.NET/Go）。
- **成熟度**：Apache-2.0 / Python ~44% / server 0.2.1（2026-06-29，149 个 release）/ ~11.7k⭐ / OpenSSF 徽章 / CNCF Landscape。
- **借鉴价值（两者中最佳设计参考）**：两个点直接映射到 Pi 的 `tool_call`：① **策略层**对标其**每沙箱出网控制**——钩子检查 network/bash 入参，拦截或改写不允许的目的地；② **凭据金库**模式——从工具调用里剥离密钥，仅在执行时注入。其**运行时可插拔（Sandbox Protocol）**正是 Pi"沙箱后端接口"的范本。**重型隔离场景**用 REST/Python SDK 从钩子驱动它；**轻量本地 Pi 扩展**则学其设计、用 bubblewrap 重写策略/出网/凭据三概念。
- **放在 Pi 的部位**：`.pi/extensions/sandbox/`——`tool_call` 钩子做 Policy Gate（路径/网络/MCP 范围判定），是 Execution Harness 的 Policy Gate 与 Tool Runtime Harness 的 capability sandbox 的交汇点。

### 2.2 ⭐⭐ agent-infra/sandbox（AIO Sandbox）—— 一体化 Docker（集成目标）

- **亮点**
  - **统一文件系统**横跨 浏览器/Shell/文件 操作——浏览器下载的文件，Shell/File 工具立刻可见，无需状态同步胶水。
  - 一个盒子里多套人机界面：VNC、VSCode Server、Jupyter、终端。
  - Agent-ready：预接 MCP servers（Browser/File/Shell/Markitdown）+ "零配置"。
- **核心机制**：单个 **Docker 容器**打包完整工作区，隔离边界=容器本身（无 microVM/gVisor 层，共享内核）；浏览器走 VNC+CDP，Shell 走 WebSocket。
- **接口**：REST API（`/v1/`）+ MCP servers + SDK（Python `agent-sandbox` / TS `@agent-infra/sandbox` / Go）。
- **成熟度**：Apache-2.0 / Python 55.8% TS 37.2% / v1.11.0（2026-06-23）/ ~5.3k⭐ / 有 arXiv 论文。
- **借鉴价值（更偏"集成目标"而非设计参考）**：最干净的用法——Pi 扩展用 `tool_call` 把 bash/file/browser 工具调用**改写成对 AIO 容器的 REST/MCP 调用**，让 agent 的文件系统和 Shell 在盒子**里**执行而非宿主机：Pi 管编排，AIO 提供牢笼。**概念借鉴**：统一文件系统契约（一个 mount 被所有工具共享）、每种能力暴露为 MCP 工具。**注意**：纯 Docker 共享内核、隔离较弱——适合可信开发 harness，不适合不可信多租户代码。
- **放在 Pi 的部位**：作为 Tool Runtime Harness 的**可选执行后端**（尤其要做 Browser Use / Agentic UI 验收时，它自带 Playwright 风格浏览器能力）。

> **沙箱选型建议**：
> - **简历要"会设计安全"** → 学 OpenSandbox 的策略/出网/凭据/可插拔运行时，自己用 bubblewrap 写一个轻量 Pi 沙箱扩展（最能体现工程能力）。
> - **要快速演示浏览器验收/重隔离** → 直接集成 AIO Sandbox 或 OpenSandbox 当后端。
> - 对齐 [Pi创新改造参考项目.md](Pi创新改造参考项目.md) 里原本列的 OpenHands sandbox，这两个项目（尤其 OpenSandbox）是**更新、更聚焦"sandbox runtime"本身**的替代参考。

## 3. 自进化 & Prompt 缓存类（落地 Skill Memory + Context Cost Harness）

### 3.1 ⭐⭐⭐ flightlesstux/prompt-caching —— 按语义角色自动缓存（省钱直接落地）

- **亮点**
  - 暴露 `optimize_messages` 工具，按**语义角色**给稳定前缀打 `cache_control`：① system prompt、② tool 定义、③ 超阈值（默认 1024 token）的大消息。
  - **零配置**（可选 `.prompt-cache.json` 调 `minTokensToCache` 等），默认即用。
  - 用**基于角色的分类**而非脆弱的动态前缀匹配；遵守 Anthropic 最多 4 个 breakpoint 的限制。
- **核心机制**：稳定性按内容类型假定（system/tools/大块在多轮间不变），形成与 Anthropic 服务端前缀缓存（5 分钟 TTL，可选 1 小时）对齐的天然缓存边界。省钱数学：缓存写 1.25× 输入、缓存读 0.1×（即省 90%），第 2 轮即回本。
- **形态**：MCP server，npm 包 `prompt-caching-mcp`。
- **成熟度**：MIT / TypeScript 70.5% / v1.3.0（2026-03-13）/ ~128⭐。
- **借鉴价值（`before_provider_request` 的最干净落地）**：复刻为 Pi 扩展——发送 payload 前 ① 把 prompt 排成 **稳定→易变**（system → tools → 持久上下文 → 当前回合），② 在最后一个稳定块边界插 `cache_control`（封顶 4 个）。**关键前缀稳定性技巧**：绝不把易变数据（时间戳/RNG/回合计数）混进缓存前缀；tool 定义每会话只缓存一次；只标 >~1024 token 的大块以免小块写开销；保持各轮前缀字节级一致让缓存真正命中。
- **放在 Pi 的部位**：`.pi/extensions/cache/`——订阅 `before_provider_request` 重排 payload + 插 breakpoint；配合 `after_provider_response` 记录 cache read/write token 做命中率仪表盘。这是 Context Cost Harness 的核心起手式。

### 3.2 ⭐⭐⭐ alchaincyf/darwin-skill —— 评估→改进→测试→保留/回滚（自进化闭环范本）

- **亮点**
  - **棘轮规则（Ratchet）**："只保留可测量的改进，其余全部回滚。"基线锁在历史最佳分；单轮收益 <1 分则早停。
  - **多评估器去偏**：改完 `SKILL.md` 后 `git commit`，**起 2 个独立子代理重新打分**（SkillLens 论文发现 LLM 自评仅 ~46% 准）。保留条件=新分 > 旧分，否则 **`git revert`**（特意不用 `git reset --hard`）。
  - v2.0 升级维度：**失败机制编码**、**可执行具体性**（禁止"考虑/建议"等模糊词）、**高危动作黑名单**（禁 `rm`/`git reset --hard`/force-push）。
  - 9 维度 100 分制评分（结构维度静态分析，效果维度用 `test-prompts.json` 真实跑）。人在环检查点。
- **核心机制**：5 阶段循环（基线评估 → 选最低分维度做单点改进 → 多评估器复评 → 保留/回滚 → 回归测试门禁），灵感来自 Karpathy 的 `autoresearch`。
- **形态**：Claude Code Skill（`npx skills add alchaincyf/darwin-skill`）。
- **成熟度**：MIT / 技能内容简体中文 / v2.0（2026-05-28）/ ~4.4k⭐。
- **借鉴价值（可直接复刻的自进化闭环）**：保留 rubric + `test-prompts.json`；每轮让 Pi 编辑技能，再 **spawn N 个独立打分子代理**；以分差 gate 保留/回滚，配 git-revert 兜底 + <1 分早停。**棘轮 + 独立评审**与 `registerTool`（通过验证后才注册改进的技能）和 `session_before_compact`（蒸馏哪些编辑有效）天然配对。
- **放在 Pi 的部位**：Skill Memory Harness 的"自进化 Skill Factory"；其多评估器子代理同时是 Eval Collaboration Harness 的雏形（独立 judge 子代理）。

### 3.3 ⭐⭐ EvoAgentX/Awesome-Self-Evolving-Agents —— 自进化分类法综述（理论弹药）

- **亮点**
  - 自进化 Agent **四轴分类法**：① 单代理优化（LLM 行为 / Prompt / 记忆 / 工具）② 多代理优化（自动构建 + MAS 优化）③ 领域特定优化 ④ 评估。

## 6. 落地建议（按你的处境）

1. **先做最容易出成果且面试一眼懂价值的三件套**：
   - **Context Cost / 缓存**（prompt-caching → `before_provider_request`）——代码量小、效果可量化（省 token %），**最快见效**。
   - **记忆**（mempalace 本地引擎 + claude-mem 钩子骨架 → `session_before_compact`/`context`）——解决"失忆"痛点，亮点强。
   - **权限沙箱**（OpenSandbox 设计 + bubblewrap → `tool_call` block）——安全焦虑刚需，价值直白。
2. **自进化作为差异化亮点**（darwin-skill 的棘轮+独立评审 → Skill Factory），但要讲清边界避免被质疑"噱头"。
3. **全程拿 learn-claude-code 当实现蓝本**，拿 `基础版本PI.md` 当"挂在 Pi 哪里"的对照表，两者交叉使用。
4. **沙箱二选一**：要体现工程深度就自己用 bwrap 重写 OpenSandbox 的策略/出网/凭据；要快速演示浏览器验收就直接集成 AIO Sandbox 当后端。

## 测评和落盘部分

我会先按三个方向查：trace/执行轨迹落盘、agent eval 框架、eval 后的记忆/技能自进化闭环。重点会筛掉只有概念没有工程形态的项目，保留能直接借鉴到 Pi-AgentX 的实现模式。

我大概查了一圈，结论是：Pi-AgentX 这块可以拆成三层做，而不是直接押宝某一个平台。

```text
Trace 落盘层：OpenTelemetry / OpenInference 语义规范
Eval 层：golden dataset + deterministic scorer + LLM judge + CI gate
Evolution 层：从 trace/eval 里蒸馏 skill / memory / prompt patch，棘轮保留
```

**1. Trace 落盘：最佳范式是 OTel + OpenInference，而不是自己发明日志格式**

目前比较成熟的方向是：

```text
Agent Run span
  ├─ Planner span
  ├─ LLM span
  ├─ Tool span
  ├─ Retriever span
  ├─ Guardrail span
  ├─ Evaluator span
  └─ Memory write span
```

每个 span 都带结构化字段：

```json
{
  "trace_id": "...",
  "span_id": "...",
  "parent_span_id": "...",
  "span_kind": "LLM|TOOL|AGENT|CHAIN|RETRIEVER|EVALUATOR",
  "name": "invoke_model",
  "input": {},
  "output": {},
  "metadata": {},
  "token_count": {},
  "cost": {},
  "latency_ms": 123,
  "status": "ok|error"
}
```

值得重点看这些项目：

**⭐⭐⭐ Langfuse/langfuse**
它是开源 LLM observability + eval 平台，支持 traces、prompt management、datasets、LLM-as-judge、user feedback、自托管，底层强调 OpenTelemetry 集成和 agent trace 可视化。它的文档明确把 agent eval 分成 final response、trajectory、single step 三类，这个分类对 Pi 的测评设计很有用。[1][2]

借鉴点：

```text
trace → score → dataset → experiment → regression gate
```

Pi 可以不用一开始接 Langfuse 服务，但 trace schema 应该兼容它。

**⭐⭐⭐ Arize-ai/openinference + Arize Phoenix**
OpenInference 是“AI 应用的 OpenTelemetry 语义约定”。它定义了 LLM、TOOL、AGENT、RETRIEVER、EVALUATOR 等 span kind，并支持 Python / JS / Go 多语言 instrumentation。[3]
Phoenix 则是自托管 trace/eval 后端，适合当本地或私有部署的 trace viewer。[5]

借鉴点：

```text
不要只记录最终回答
要记录每一次 LLM call、tool call、retrieval、judge、memory write
```

尤其注意一个实际坑：只包最外层 agent run 不够。AutoGen 相关 issue 里有人指出，如果没有在 OpenAI SDK call boundary 做 per-call instrumentation，就会丢失 messages、params、tokens、tool-calls 等关键细节。[1]

所以 Pi 的 trace 采集应该放在这些边界：

```text
provider request boundary
tool invoke boundary
subagent spawn boundary
memory read/write boundary
eval scorer boundary
file edit / shell command boundary
```

不是只在 “runAgent() 开始/结束” 记一条日志。

**推荐 Pi 落盘方案**

先做轻量本地版：

```text
.pi/
  traces/
    runs.sqlite
    raw/
      2026-xx-xx/<trace_id>.jsonl
  evals/
    datasets/
    runs/
  memories/
    skills/
    lessons/
```

SQLite 表：

```text
agent_runs
spans
span_events
tool_calls
llm_calls
eval_scores
memory_candidates
evolution_patches
```

同时每个 trace 额外写一份 append-only JSONL，方便调试、回放、导出：

```json
{"type":"span_start","trace_id":"...","span_id":"...","kind":"LLM","name":"provider_request"}
{"type":"span_end","span_id":"...","output":{},"tokens":{},"status":"ok"}
{"type":"score","trace_id":"...","metric":"tool_correctness","score":0.86}
{"type":"memory_candidate","trace_id":"...","kind":"good_pattern","text":"..."}
```

这样 Pi 不会一开始被 Langfuse/Phoenix/Braintrust 绑定，但以后可以导出 OTLP 或 OpenInference 格式。

---

**2. 测评：不要只做 LLM judge，要做“轨迹级测评”**

测评部分建议分四种：

```text
black-box eval：只看最终答案
trajectory eval：看工具调用轨迹是否正确
step eval：看每一步工具/LLM 调用是否合理
regression eval：新版本不能比历史最佳差
```

这个分类和 Langfuse 的 agent eval 思路一致：final response、trajectory、single step 三种评估方式。[1]

可借鉴项目：

**⭐⭐⭐ promptfoo/promptfoo**
适合做 prompt、agent、RAG 的自动化 eval 和 red team，CLI + 配置驱动，支持 CI/CD，本地运行友好。[1]
Pi 可以借它的思路做：

```yaml
tests:
  - vars:
      task: "修复一个 failing test"
    assert:
      - type: contains
      - type: javascript
      - type: llm-rubric
```

适合放在：

```text
Pi Eval Harness
```

**⭐⭐⭐ DeepEval**
Python 生态里更像 pytest 风格的 eval。适合把 agent eval 写成测试，跑在 CI 里。它强调 Task Completion、Tool Correctness、RAG faithfulness、hallucination/bias/toxicity 等指标。[2][3]

Pi 如果后端偏 TypeScript，也仍然可以借它的结构：

```text
test case
metric
scorer
threshold
eval run
CI gate
```

**⭐⭐ Inspect AI**
更偏安全、公共部门、多 provider benchmark。它适合做严肃 benchmark 和安全测评参考，不一定直接接入 Pi。[2][4]

**⭐⭐ Braintrust / LangSmith / Phoenix / Opik**
这些偏完整平台。Braintrust 强在 dataset-first、CI quality gate、trace + eval + prompt 优化闭环；Phoenix 强在 OTel-native、自托管；LangSmith 强在 LangChain/LangGraph 深集成。[2][3]

Pi 的实用选择：

```text
本地第一阶段：SQLite + JSONL + promptfoo-like YAML eval
后续增强：OpenInference OTLP export → Phoenix/Langfuse
CI 阶段：eval score threshold + regression diff
```

**3. 测评后自动进化：可以做，但必须有闸门**

你说的“好的 trace 沉淀，坏的修改”，我建议拆成两个队列：

```text
Good Trace → Skill / Memory / Workflow Pattern
Bad Trace  → Failure Mode / Regression Test / Prompt Patch Candidate
```

不要让 agent 直接“看到坏 trace 就改自己”。正确闭环应该是：

```text
1. trace 落盘
2. eval 打分
3. 失败归因
4. 生成 memory/skill/prompt patch candidate
5. replay eval
6. 分数提升才合并
7. 否则丢弃或保留为 negative example
```

这和前面 darwin-skill 的 ratchet 思路一致。

可参考方向：

**Reflexion**
核心是 agent 从失败里写 verbal reflection，后续任务复用。适合 Pi 的 bad trace → lesson。[3]

**Voyager / Skill Library**
成功轨迹沉淀成可复用 skill。适合 Pi 的 good trace → skill file。[1][3]

**Acontext / agentmemory / Mem0 / Letta**
这些偏 memory system。尤其 Acontext 这种“从 agent runs 自动捕获 learnings 并存成 reusable skill files”的方向，和 Pi 的 Skill Memory Harness 很贴。[3]

**Self-Evolving Agents 综述类仓库**
自进化 agent 的研究分类里明确有 dynamic experience evolution、skill augmented evolution、memory architecture evolution、tool-augmented evolution 等轴线，刚好对应 Pi 的 trace 后处理模块。[1][3]

但这里有一个安全点必须写进设计：
memory/skill 自动沉淀会带来 poisoning 风险。AgentPoison 专门研究了通过 memory 或 knowledge base 投毒攻击 agent 的方式。[4]
所以 Pi 的 memory 写入必须有：

```text
source trace id
eval score
human approval flag
expiration / rollback
poisoning scan
scope: project/user/global
```

**我建议 Pi-AgentX 这样落模块**

```text
Trace Harness
  负责记录完整 agent 执行轨迹
  输出 OpenInference-compatible spans + 本地 SQLite/JSONL

Eval Harness
  负责把 trace 跑 scorer
  支持 deterministic scorer / LLM judge / human label / regression gate

Trace Distiller
  从高分 trace 提炼 good pattern
  从低分 trace 提炼 failure mode

Skill Memory Harness
  good trace → skill candidate
  bad trace → anti-pattern / test case / prompt patch candidate

Evolution Gate
  replay eval
  新分 > 旧分才保留
  否则 reject/revert
```

最小可行闭环：

```text
run task
→ save trace
→ eval trace
→ if success/high score: distill reusable skill
→ if failure/low score: create regression test + failure note
→ candidate patch
→ replay eval
→ pass threshold 才 registerTool/registerSkill
```

**最值得直接复刻的实现路线**

第一阶段不要上复杂平台，先自己实现：

```text
.pi/traces/runs.sqlite
.pi/traces/raw/*.jsonl
.pi/evals/test-prompts.json
.pi/evals/rubric.json
.pi/memory/skills/*.md
```

第二阶段兼容：

```text
OpenInference semantic attributes
OTLP exporter
Langfuse/Phoenix import
promptfoo-style eval config
```

第三阶段做自进化：

```text
good_trace_distiller
bad_trace_failure_encoder
skill_candidate_generator
ratchet_eval_gate
git revert fallback
```

一句话：
Pi 这里最优设计不是“买一个 observability 平台”，而是内部用 OpenInference/OTel 风格把 trace 结构化落盘，再用 eval 把 trace 变成可评分资产，最后只把通过回放测评的经验沉淀成 skill/memory。好的 trace 进 skill factory，坏的 trace 进 failure library 和 regression dataset，所有自修改都走 ratchet gate。
