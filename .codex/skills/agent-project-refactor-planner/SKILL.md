# Agent 项目改造规划

## 适用范围

给一个**已存在的开源 agent 领域项目**（编码 agent、agent harness、agent runtime、记忆/工具/评测框架等）按自己的想法做改造，且当前处于**只讨论/写文档、暂不写代码**的规划阶段时使用。

产出是「改造方案文档」，不是代码。目标：把「在别人的底座上加自己的治理/能力层」这件事，讨论清楚、决策留痕、能对齐下一个人（或下一个 AI）。

不适用：从零起项目、纯 bug 修复、已进入编码实现阶段（那时改用具体实现流程）。

## 核心原则（为什么这套方法有效）

1. **先摸底座，再设计**——agent 项目的价值在于「补底座缺的那部分」，不是重造轮子。设计任何模块前，先查清「底座已经实现了什么、源码层面怎么做的」。很多设想的功能底座早已做掉（例：某编码 agent 已自动做 prompt 缓存、已做 skill 渐进披露）。
2. **查证不猜**——涉及底座能否做某事（能否拦截工具、能否改 payload、能否回退）时，**读源码确认**，不靠记忆或钩子名猜。一次「以为能在 A 钩子做、实则不行」的返工，比先查证贵得多。
3. **逐模块、逐子议题讨论**——一次只推进一个模块的一个子议题；每个子议题给用户 2-3 个带取舍的选项让其拍板，不替用户做架构决策。
4. **每个决策写明理由**——不只写「做什么」，写「为什么这样而不是那样」（含被否方案、依据的事实）。
5. **给演化路线**——每个模块写 V1（现在做什么）+ V2/V3（将来加什么、为什么推后）。避免 V1 塞太多，也让读者看到全貌。
6. **单一进度事实源**——维护一份进度表（模块总览 + 每模块子议题勾选 + 当前指针），每次对齐先看它。跨会话/跨 AI 靠它续上。
7. **保守默认 + 先观测后自动化**——治理类模块（拦截/审批/评分）默认取保守姿态；能力先做「观测/采集」，把「自动决策」留给后续版本。

## 工作流

### 第 0 步：建立事实基础

- 确认工作目录、仓库、底座项目位置（底座常作为只读参考子目录，勿改）。
- 找到并读：项目的模块索引/backlog、已有的设计 stub、以及**开源参考项目调研**（若有）。
- 校准现实与旧上下文的差异——目录可能被改名、某功能可能已实现非骨架、路径可能变。**以磁盘真实代码为准**，不以接力文档为准。差异要当面跟用户确认，别默默假设。

### 第 1 步：锁定全局模型与模块栈

- 和用户确认核心概念模型、模块清单、实现顺序、跨模块约束（如数据写哪、什么算关键事件）。
- 这些是所有模块的硬约束，先钉死再逐个展开。

### 第 2 步：逐模块讨论（每模块循环）

对每个模块：

1. **摸底座**：通常派一个只读探索 agent 查源码，问准「这个能力底座有没有、怎么实现的、扩展挂点在哪、能不能 mutate/block/回退」。把结论存进记忆（下个模块可能复用）。
2. **划 V1 范围**：一个模块常横跨多件复杂度递增的事（如 记忆/路由/自进化）。先和用户定 V1 做哪些、哪些推后，理由讲清。
3. **逐子议题拍板**：列子议题清单，一项项走。每项用 2-3 个带取舍选项问用户（可用带 preview 的选择）。定一个，归档一个到进度表。
4. **落最终方案文档**：全部子议题定完后写，结构见下。

### 第 3 步：写最终方案文档（每模块一份）

固定结构（可按项目微调）：

- **一句话定位** + V1 灵魂
- **底座现状 → 本项目增量 → 实现前后区别**：① 底座已实现什么、源码怎么做的（带文件行号）；② 本模块架构；③ 我们还要实现什么、如何实现；④ 实现前后行为/能力有何区别（对比表最直观）。
- **决策（含理由）**：每条「选择 + 为什么这样而不那样」。
- **实现**：钩子/挂点表、关键流程、落点。
- **演化路线**：V1 / V2 / V3 表。
- **架构优点**（几条）
- **少量缺点**（诚实列，含协调点/技术债）

### 第 4 步：维护进度表与记忆

- 进度表：模块总览表（状态图例 ✅🟡🔄⬜ + 👉当前指针）+ 每模块子议题清单 + 已查证事实 + 全局决策。
- 记忆：项目上下文、底座能力查证结果、每模块决策摘要、文档规则。跨会话靠记忆 + 进度表续上。
- 收尾时把「下一步从哪继续 + 待答问题」写进进度表指针，方便下次直接接。

## 关键手法

- **派探索 agent 查源码**：问题要具体到「返回什么值能 block」「event.input 能不能改」「哪个钩子能过滤 X」，并要求带文件行号引用、找不到就明说不猜。
- **决策问题给可对比选项**：每个 AskUserQuestion 的选项写清取舍（推荐项标注 + 一句话说清代价），必要时附 preview 做可视对比。
- **归档即时化**：每定一个子议题，立刻更新进度表对应行 + 指针，别攒着。
- **跨模块判据复用**：某模块确立的判断标准（如「什么事件写主线、什么只进日志」），后续模块直接套用并在进度表标注「可复用」。

## Update Rule

每当发现新的坑、更好的做法、或用户提出新的文档/工作规则时，**在同一会话内更新本 skill**，并单独 commit 这次更新。

## 本项目（EvoPi）实战教训

- 底座（Pi 编码 agent）**已自动做 Anthropic prompt 缓存**（占了 4 个 cache_control 断点里的 3 个）——手动再加是重复且有害。→ 先查底座缓存机制再设计 Cost 模块。
- 原以为能在 `context` 钩子做 skill 的 Top-K 路由，**查源码发现不行**（系统提示在该钩子前已固定）——正确挂点是 `resources_discover`。查证救回了一次错误设计。
- 底座的 `tool_call` 钩子**能真 block**（返回 `{block,reason}` 转成错误结果给模型）、`event.input` 可原地改、`ctx.ui` 有 confirm/select/input——执行治理模块技术上全可行，查实后才敢定「主动拦截」。
- 用户会中途重组目录（`docs/evopi/` → `docs/evopi-v1/`）或留「小记」提新要求——**每次开工先核对磁盘真实结构**，把新规则并入文档规则。
- 「项目受信任」≠「可以执行危险操作」——治理模块要区分这类语义，别混用底座的信任开关。
- 保守默认 + 先观测后自动化：Cost 先观测缓存命中率（V2 才优化）、记忆自动抽取先做候选（人审后生效）、验收先摆证据（V2 才自动裁决）、rewind 先手动（V2 才自动）。这个节奏反复出现，是可靠的 V1 收敛法。
