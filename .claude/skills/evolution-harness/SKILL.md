# /evolution-harness

把一个**已经跑起来**的项目，安全地带入**持续演化阶段**。解决一个真实痛点：**新输入不断进来（文章 / 开源项目 / 灵感 / "这里不优雅"），怎么让它们有序改造项目，而不是一有想法就乱、计划天天变。**

核心手段 = **捕获与评判分离（灵感池）+ 五步漏斗（消化）+ 决策留痕（ADR）+ 路线图（排期）**。

> 本 skill 目录（下称 `<SKILL_DIR>`）随附可复用模板，执行时按相对路径读取：
> - `templates/演化流程.md` — 五步漏斗 SOP（骨架），几乎可直接落地
> - `templates/灵感池.md` — 捕获层模板（含条目模板 + 状态流转）
> - `templates/ADR-模板.md` — 架构决策记录模板
> - `templates/路线图.md` — 候选功能 + 优先级模板

---

## 适用前提（不满足就先别用）

- **项目已过"从零实现"阶段**：至少有一个能跑的版本。还在从设计到首版实现的路上 → 先用 `impl-harness`。
- **预期会长期迭代**：会不断有新输入想并进来。一次性交付、不再动的项目不需要。
- **单人或小团队**：本 skill 刻意轻量（几个 markdown + 一个漏斗）。大团队要正式 RFC/看板流程的，另配重型工具。

---

## 为什么需要它（讲给使用者）

已跑起来的项目最容易死在两个极端：

1. **来者不拒**：读一篇文章就想重构一次，计划天天变，什么都做不完。
2. **来者不理**：埋头做既定计划，好点子当场拒掉或忘掉，项目停止进化。

这套 harness 是中间道：**任何新输入都先无门槛地"接住"，再在固定节奏里冷静"消化"**——够格的才动手，不够格的留痕存档（避免下次又捡起来重新纠结）。

## 五步漏斗（核心）

```text
新输入（文章 / 开源 repo / 自己的灵感 / "这里不优雅"）
   │
   │  ①  捕获 —— 一行字先记进 灵感池，不评判、不打断手头的事
   ▼
   ②  提炼 —— 有空时回看：它到底主张什么？对本项目哪个部分有可借鉴点？
   ▼
   ③  定性 —— 归到四类之一：
   │      ├─ A 新功能      → 进 路线图 排优先级
   │      ├─ B 架构优化    → 写一篇 ADR，评估「爆炸半径」
   │      ├─ C 推翻旧决策  → 写 ADR + 更新既有的冻结契约/变更日志（记清"为什么改"）
   │      └─ D 不适合      → 灵感池里标 ✗ + 一句理由（"看过，不用，因为___"）
   ▼
   ④  排期 —— 够格的进路线图标优先级；不够格的留灵感池
   ▼
   ⑤  实施 —— 走项目既有的实现流程（如 impl-harness：冻结契约→拆文件→自测→小步提交）
```

## 三条要害规则（让它"不乱"的关键）

### 规则 1：捕获 ≠ 评判（时间上分开）
灵感来了**只记不判**。当场决定"做不做"要么打断手头的事，要么草率拒掉好点子。捕获是 10 秒的事，评判留到有整块时间时集中做。

### 规则 2：决策必须留痕（ADR）
任何"要不要改架构 / 要不要推翻旧决定"，写一篇简短 **ADR**：背景、备选、决定、理由、影响。
- 给未来的自己：三个月后会忘"当初为什么这么设计"。
- 给评审/面试：ADR 展示的是"会演化系统、会权衡、改动前会评估影响"——比"会写代码"值钱。
- 约束冲动：写 ADR 逼你把"我想改"变成"我为什么改、代价是什么、有没有更小的做法"。

### 规则 3：契约可解冻，但要走流程
项目若定过冻结约定（架构铁律），**不是不能改**，而是：
- 先问：这是**加法**（加字段/加模块，向后兼容）还是**改造**（改名/删字段/改结构，牵动下游）？
- **加法**：低风险，直接做，进度表加一行。
- **改造**：先写 ADR 评估爆炸半径，再动。**加法优先**——能用"加新东西+逐步迁移"达成的，不要"原地改结构"。

### 「爆炸半径」怎么估（改造类专用）
改一处前问三个问题，越多"是"越危险、越该先写 ADR：
1. 有没有下游依赖它的稳定形态？（如数据格式被存储/看板/自动化依赖）
2. 动它要不要同时改多个模块？（如动底座 → 所有消费者受影响）
3. 它是不是一条已冻结的全局约定？

## 执行步骤（skill 被调用时怎么落地）

1. **定目录**：与用户确认演化文档放哪（默认 `docs/<项目名>-evolution/` 或版本目录 `docs/vN/`）。
2. **铺四份文件**（从 `<SKILL_DIR>/templates/` 拷贝并按项目改写）：
   - `演化流程.md`（漏斗 SOP，骨架几乎可直接用，补项目专属的"实施流程"链接）；
   - `灵感池.md`（捕获层；**建议立刻放 3-5 条种子**——从项目既有的"待办/下一步/调研"里提炼，别留空文件）；
   - `路线图.md`（把种子里定性为 A 的排优先级）；
   - `架构决策记录/`（README + ADR 模板；若项目当下有真实待决策，写第一篇 ADR 兼做示范）。
3. **接既有资产**：演化流程的"实施"一步要指向项目已有的实现流程（如 impl-harness）；"提炼"一步可复用项目已有的调研范式。**不另起炉灶。**
4. **回写自身**：用完这套后，若发现漏斗/规则有可改进处，回写本 skill（与 impl-harness 同哲学：越用越准）。

`--scaffold-only`：只铺文件骨架，不放种子、不写首篇 ADR。

---

## 一句话 SOP（贴给使用者）

> **想法来了 → 记进灵感池（别停下手头的事）。有空了 → 提炼、定性（A新功能/B架构/C推翻/D不用）。够格的 → 进路线图或写 ADR。动手时 → 走项目既有实现流程。改架构/推翻旧决定 → 先写 ADR 估爆炸半径。**

---

## 与 impl-harness 的关系

- `impl-harness`：管 **"方案已定 → 首版实现"**（防漂移/防失忆/控 token）。
- `evolution-harness`（本 skill）：管 **"首版实现后 → 持续演化"**（新输入怎么有序并入）。
- 两者接力：impl 阶段的"变更日志"自然升级为演化阶段的"ADR"；impl 的实现流水线成为演化漏斗第⑤步的"实施"。
