# <项目名> 演化流程（新输入怎么变成改动，而不把项目搅乱）

> 本项目已过"从零实现"阶段。这份文档解决下一个问题：**新输入不断进来（读了文章 / 看了开源项目 / 冒出灵感 / 觉得某处架构不优雅），怎么让它们有序地改造项目，而不是一有想法就乱。**
>
> 一句话原则：**捕获与评判分离，决策留痕，契约可解冻但要走流程。**

## 五步漏斗

```text
新输入（文章 / 开源 repo / 自己的灵感 / "这里不优雅"）
   │
   │  ①  捕获 —— 一行字先记进 灵感池.md，不评判、不打断手头的事
   ▼
   ②  提炼 —— 有空时回看：它到底主张什么？对本项目哪个部分有可借鉴点？
   ▼
   ③  定性 —— 归到四类之一：
   │      ├─ A 新功能      → 进 路线图.md 排优先级
   │      ├─ B 架构优化    → 写一篇 ADR，评估「爆炸半径」
   │      ├─ C 推翻旧决策  → 写 ADR + 更新既有冻结契约/变更日志（记清"为什么改"）
   │      └─ D 不适合      → 灵感池里标 ✗ + 一句理由（"看过，不用，因为___"）
   ▼
   ④  排期 —— 够格的进路线图；不够格的留灵感池
   ▼
   ⑤  实施 —— 走项目既有的实现流程 <这里填：如 impl-harness / 你的开发流程>
```

| 步 | 动作 | 落在哪 | 花多久 |
| --- | --- | --- | --- |
| ① 捕获 | 记一行 | 灵感池.md | 10 秒 |
| ② 提炼 | 写两三句"主张什么/可借鉴点" | 灵感池.md 同条下 | 几分钟 |
| ③ 定性 | 打 A/B/C/D 标签 | 灵感池.md | 1 分钟 |
| ④ 排期 | A/B/C 升级到路线图或 ADR | 路线图.md / 架构决策记录/ | 几分钟 |
| ⑤ 实施 | 编码 | 代码 + 提交 | 按模块 |

## 三条要害规则

### 规则 1：捕获 ≠ 评判（时间上分开）
灵感来了只记不判。当场决定"做不做"会打断手头的事、或草率拒掉好点子。捕获 10 秒，评判留到有整块时间时集中做。

### 规则 2：决策必须留痕（ADR）
任何"改架构 / 推翻旧决定"，写一篇简短 ADR：背景、备选、决定、理由、影响。给未来的自己、给评审、也约束自己的冲动。

### 规则 3：契约可解冻，但要走流程
若项目定过冻结约定：改动前先分清**加法**（向后兼容，直接做）还是**改造**（牵动下游，先写 ADR 估爆炸半径）。**加法优先。**

### 「爆炸半径」怎么估（改造类专用）
改一处前问三个，越多"是"越危险、越该先写 ADR：① 有下游依赖它的稳定形态吗？② 要同时改多个模块吗？③ 是已冻结的全局约定吗？

## 一句话 SOP

> 想法来了 → 记进灵感池（别停手头事）→ 有空提炼定性（A/B/C/D）→ 够格的进路线图或写 ADR → 动手走既有实现流程 → 改架构先估爆炸半径。
