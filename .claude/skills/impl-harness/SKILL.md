# /impl-harness

把一个**方案已定稿**的项目，安全地带入**实现阶段**。解决四个真实痛点：**任务漂移**（做着做着偏离方案）、**跨窗口失忆**（新会话/压缩后忘了做到哪、忘了铁律）、**大文件吃 token**（文档与代码重复膨胀）、**质量与可扩展**（词表/挂点/验收不稳导致返工）。

核心手段 = **进度表（防漂移）+ hook（防失忆）+ 拆文件约定（控 token）+ 小步流水线（保质量）+ 自回写（越用越准）**。

> 本 skill 目录（下称 `<SKILL_DIR>`）随附可复用模板，执行时按相对路径读取：
> - `templates/inject-progress.mjs` — SessionStart 注入进度关键节的 hook 脚本模板
> - `templates/snapshot.mjs` — PreCompact 落黑匣子快照的 hook 脚本模板
> - `templates/settings.hooks.json` — 把上面两个 hook 挂上的 settings 片段

---

## 适用前提（不满足就先别用）

1. **方案文档已定稿**：每个模块/组件的设计已经写下来（挂点、数据结构、事件/接口 schema、边界）。本 skill **不做设计**，它把"已定的设计"变成"能跑的实现骨架 + 推进节奏"。
2. **是渐进式、多模块、可能跨多个会话的编码任务**——一次会话就能写完的小改动不需要这套 harness（那属于过度工程）。

若方案还没定稿 → 先去做方案讨论，别套这个 skill。

---

## 参数解析（`$ARGUMENTS`）

- **方案文档目录**（第一个非 `--` 参数）：如 `docs/evopi-v1/`。解析为 `$DOCS`。默认让用户指认。
- `--no-hook`：不配 hook，只建进度表+规约（用户环境不便用 hook 时）。
- `--scaffold-only`：只搭脚手架，不进入模块实现循环（搭完停下让用户决定）。

---

## 贯穿始终的三条铁律（skill 自身也遵守）

1. **同一件事不写两遍**：设计信息（schema/挂点/逻辑）只在方案文档里；实现阶段**不重写**成第二份 impl 文档，只在进度表里补"实现清单 + 验收判据"。重复是大文件与脱节的根源。
2. **机器管"不丢"，我管"写对"**：hook 只做机械的读入与快照，**绝不自动改进度表**（脚本不理解"做到哪了"，让它猜会污染事实源）；进度表的语义更新由执行者手动做。
3. **小步**：每个可验收单元一做完立刻"勾进度表 + 提交"，把"未落盘的进行中工作"压到最小——这是防失忆的核心纪律，不只是 git 卫生。

---

## 阶段 0 — 对齐（先读，别急着建文件）

1. 读 `$DOCS` 下的方案文档与任何现有进度/架构文档，摸清：模块清单、依赖顺序、每模块的挂点与事件/接口 schema、已锁定的全局约定、代码落点。
2. 若方案文档里**已含**挂点/schema/逻辑/数据结构 → 确认它们，**准备"冻结"当契约**，不重写。
3. 若缺**验收判据**（"怎么算这个模块做完"）→ 这是唯一要补的东西，阶段 1 写进进度表。
4. 用 AskUserQuestion 跟用户确认三件可能有歧义的事（若方案已写明则跳过）：实现文档形态（推荐"不重写，只补清单+验收"）、防失忆机制（推荐"进度表+hook"）、控 token（推荐"文件行数上限+拆分"）。

## 阶段 1 — 建实现进度表（单一事实源，防漂移的锚）

在 `$DOCS/impl/进度.md`（或项目习惯位置）建实现进度表，**必须包含**：

- **顶部「冻结的全局约定」节**：把跨模块、实现期不得漂移的铁律集中列出（事件词表冻结、共享定义单一事实源、安全/资源优先级、保守默认等）。这是 hook 每次要注入、跨窗口最该被反复看见的内容。**加字段可以（向后兼容），改名/删字段要记"变更日志"。**
- **模块实现总览表**：每模块拆成固定的可勾选步骤列（如：契约/接线/逻辑/命令/验收），每步一个状态格（⬜/🔄/✅），并标出当前 👉。
- **各模块「实现清单 + 验收」节**：契约（挂点/schema）**引用方案文档、不重写**；只写"拆哪些文件"+"验收判据（跑什么命令、看到什么、什么算 pass）"。**未开始的模块只留标题，待轮到再展开**（避免一次写一大坨）。
- **「变更日志」表**：改动冻结约定必须记一行（含理由）。
- **「待办指针」节**：一句话说清"当前在哪、下一步做什么"——hook 会注入它。

同时建 `$DOCS/impl/README.md`（实现规约）：把上面三条铁律 + 控-token 四约定（单文件软上限~500行 / 大产出只落数据目录不进上下文 / 读文件优先分段 / 文档不与代码重复）+ 拆文件规范 + 每模块流水线 + 提交约定，写成一页操作手册。

## 阶段 2 — 配防失忆 hook（除非 `--no-hook`）

读 `<SKILL_DIR>/templates/` 三个模板，落地到项目：

1. 把 `inject-progress.mjs` 和 `snapshot.mjs` 拷到项目的 hook 目录（如 `.claude/hooks/`，**必须是不被 gitignore 的位置**——hook 是团队基础设施要入库）。**改脚本里的进度表路径**为阶段 1 的实际路径。
2. 把 `settings.hooks.json` 的内容并入项目 `.claude/settings.json`（SessionStart→注入进度、PreCompact→落黑匣子快照到数据目录）。
3. **确认快照输出目录已被 .gitignore 忽略**（黑匣子不进版本库）。
4. **用真实 stdin 负载 pipe-test 两个脚本**（`echo '{...}' | node script.mjs`），确认 exit 0、注入内容精准、快照真落文件——漏这步 hook 可能静默失效。
5. 告知用户：若 settings.json 是本会话新建的，**hook 可能本会话未生效**，需 `/hooks` 重载或重启才触发（脚本本身已测通）。

**hook 的分工边界（重要）**：SessionStart 只注入进度表的"当前进行中 + 冻结约定 + 待办指针"几节，**不注入全文**（省 token）；PreCompact 只**追加**"时间戳+git状态+最近改动"到快照文件，**不碰进度表**。

## 阶段 3 — 提交脚手架（独立单元）

脚手架（进度表 + 规约 + hook + settings）是一个完整可验收单元 → 用项目约定的提交风格（如中文 commit）**单独提交**，与后续模块分开，历史更清晰。

若 `--scaffold-only` → 到此停下，报告交付物，问用户是否继续进入模块实现。

## 阶段 4 — 按模块实现（每模块走同一条流水线）

按方案文档的**依赖顺序**，一次一个模块：

```text
① 冻结契约  —— 引方案文档的 schema/挂点/验收，不重写
② 拆子文件  —— 新建 <模块> 源文件，源文件顶部写"本模块铁律"注释；主入口只加一行注册
③ 核心逻辑  —— 只实现方案里的 V1/当期范围（标 V2 的不做）
④ 命令 / UX
⑤ 自测验收  —— 跑命令 → grep 数据目录看到预期事件/产物 → 方案文档验收判据全绿
⑥ 提交+勾表 —— 小步提交（一个可验收单元一次），手动更新进度表对应格
```

**开工每个模块前**：先派一个只读探索代理（或自己分段读）摸清要改的文件的**代码边界与公共设施依赖**，避免迁移/改动时漏依赖或误伤其他模块。**不整文件读大文件进上下文**（用搜索/分段）。

**拆文件原则**：当期所有模块可共存于一个扩展/包，但**按模块拆成子文件**，主入口只做注册+路由；共享定义（如策略/危险动作清单）单点加载，各模块问它。避免主文件持续膨胀。

## 阶段 5 — 回写优化本 skill（自进化，每个模块做完后执行）

**这是 skill"越用越准"的关键，不可省。** 每完成一个模块（或遇到明显的卡壳/返工）后，花一分钟回顾并回写：

1. **哪里卡壳/返工了？** —— 是不是某个阶段的步骤不够具体、漏了一个前提、或某个"铁律"其实该更早出现？
2. **哪个假设错了？** —— 比如某个挂点在实操中和方案写的不一样、某个 pipe-test 命令在某平台要调整、某个"验收判据"太模糊无法判定。
3. **把结论回写到本 SKILL.md**：
   - 通用的教训 → 改进对应阶段的步骤/铁律（让下个模块、下个项目受益）。
   - 平台/环境相关的坑（如 Windows 换行、jq 不可用、hook 加载时机）→ 补进对应阶段的注意点。
   - 新发现的可复用模板 → 加到 `<SKILL_DIR>/templates/`。
4. **只沉淀"可复用的套路层"**，不要把某个具体项目的业务细节写进 skill（那属于项目的进度表/记忆，不属于 skill）。
5. 回写本身也小步：改一处、说清为什么改。若改了 skill，在给用户的回复里点明"已据本轮经验优化 skill 的 X 处"。

## 完成判据

- 进度表存在且是单一事实源；顶部有冻结约定、有变更日志、有待办指针。
- hook 已 pipe-test 通过；快照目录被 gitignore；`.claude` 可入库。
- 脚手架已独立提交。
- 进入模块循环后，每模块严格走"契约→接线→逻辑→命令→验收→提交勾表"，且做完有回写 skill 的动作。
- 全程未产生"与代码重复的第二份实现文档"，未把大产出物读进上下文。
